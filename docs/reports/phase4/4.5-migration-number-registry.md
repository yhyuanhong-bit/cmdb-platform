# Phase 4.5 — Migration 序号统一 Registry

> 关联路线图:`/cmdb-platform/docs/reports/audit-2026-04-19/REMEDIATION-ROADMAP.md` § 4.5
> 撰写日期:2026-04-20
> 适用服务:`cmdb-core` + `ingestion-engine`(跨服务治理)
> 状态:待实施,且已发现一处 **实际撞号**

---

## 目标

在两个共库微服务(`cmdb-core` 与 `ingestion-engine`)之间建立统一、可机读的 migration 序号
注册表,使"某个 3-digit `NNNNNN_` 前缀被谁占用、做什么"一目了然。同时在 CI 层加入**撞号
守卫**,让未来任何人提交同号但不同内容的 migration 时立即失败,而不是等到生产环境部署时
才发现 `golang-migrate` 的 `schema_migrations` 表报"dirty"或"相同版本号应用了不同内容"
的隐性损坏。

### 实际撞号(CRITICAL)

```
cmdb-core:        000011_prediction_tables.up.sql
ingestion-engine: 000011_bmc_field_authorities.up.sql
```

**两个服务都有 `000011_*`**,且主题完全不同。虽然 **目前两者连的是同一套 Postgres,
但用的是两套独立的 `schema_migrations` 表**(只要两个服务连接串里 `application_name`
或 `search_path` 被正确分离),因此这个撞号**尚未触发生产事故**。但它违反"同前缀即同含义"
的隐式契约,给人造成认知负担,且一旦有人把两个服务合进同一个 migrations 目录、或者用共享
的 `schema_migrations` 表重放,就会立即产生:

- `golang-migrate` 看到 version=11 已应用,跳过另一个;**数据永久缺失**
- 下一次 CI diff 会把"为什么 11 对应的 SQL 在两边不同"变成阻塞问题

### 序号分布概览

| 服务 | 序号范围 | 条数 | 缺号 |
|------|---------|-----|------|
| cmdb-core | 000001-000050(剔除后见缺号列) | 55 | **000010**、000040-000043 |
| ingestion-engine | 000010-000011 | 2 | — |

**观察**:

1. cmdb-core 的 000010 空洞 —— 事后发现 **ingestion-engine** 占用了 000010。
   这是两个服务默契地"让出"编号的产物,但**零文档记录**。新人若不读两边目录,
   会无意中重复占用。
2. cmdb-core 的 000040-000043 空洞 —— 根据路线图 § 1.2/1.3/1.4 的注释,原计划留给
   Phase 1 的 `user_roles` tenant_id、username 唯一性、inventory tenant 过滤、dedup 表。
   但实际落地时(见 commit 日志)这些需求被合并压缩进 000045-000049,40-43 号被跳过。
3. cmdb-core 的 000050 是当前末位,下一条新 migration 理应从 000052 起
   (000051 在 § 4.2 报告中预留)。

## 风险 & 触发场景

| 场景 | 现状下的后果 | 严重度 |
|------|-------------|--------|
| 两服务开发者同时新建 000052 migration | 每个服务本地 `schema_migrations` 各自前进;PR 合并时可能看似无冲突,但一旦有任何"统一 migrator"工具就撞号 | HIGH |
| 未来做统一 Postgres schema dump / restore | `golang-migrate` 看到 version=11 已应用,跳过同号文件 → 静默数据缺失 | **CRITICAL** |
| 运维把两服务合并到同一数据库 schema(例如成本压缩) | `schema_migrations` 直接产生 primary key 冲突或脏版本 | CRITICAL |
| 多人同时开 PR 创建 000052 | git merge 允许同名文件但 `make migrate-up` 时后装的一条会抛 `error: Dirty database version` | HIGH |
| Down migration 跨服务互相依赖 | 如 ingestion 的 000011 down 不知道 core 也在用 11,修 down 时破坏 core 的状态 | MEDIUM |
| 新人 onboarding | 完全找不到"下一个可用序号是多少",只能各凭感觉递增 | LOW |

## 设计决策

### 选项 1:全局顺序编号(当前隐式契约)

两个服务共用一个递增 counter。

优点:`NNNNNN_name.up.sql` 保持原格式,`golang-migrate` 无需改动。
缺点:跨服务需要协调;两人同时 +1 就撞号;没有 namespacing。

### 选项 2:按服务划分高位段(range-based prefix)

cmdb-core 占 `0NNNNN`(0-99999),ingestion-engine 占 `1NNNNN`(100000-199999),
未来服务 `2NNNNN`...

优点:物理上不可能撞号;看前缀就知道归属。
缺点:
- 要改所有现存文件名(54 个)—— **breaking**,`schema_migrations` 内容失效
- `golang-migrate` 要求版本号单调递增;改名即是改版本,会被识别为全新 migrations 并报错
- ingestion-engine 需从 000010 改到 100010,等价于回滚重建 schema

**风险太高,不采用**。

### 选项 3:保留序号 + 加 registry 文档 + CI 撞号检测

保持当前 `NNNNNN_` 命名,新增:

- `docs/MIGRATIONS.md` — 权威 registry 表格,人读
- `scripts/check-migration-collision.sh` — 机读,CI 调
- `.github/workflows/migration-check.yml` — CI 挂载点

优点:
- 零 breaking
- 对所有已应用 migration 无任何影响
- 撞号即刻被 CI block
- registry 是唯一 source of truth,新 PR 必须同时更新

缺点:
- 需要人主动维护 registry(但 CI 可反向校验缺条目 → fail)

### 选项 4:UUID / timestamp migration names

`golang-migrate` 支持非数字前缀,但会失去单调递增能力,且所有 CI 脚本、`make migrate-up`
目标都要改。ROI 太低。

### 选项 5:pgmig / atlas / flyway

换迁移工具。超出本 phase 范围。

### 决策

**选 3**:保持序号格式,补齐 `MIGRATIONS.md` 文档 + CI 撞号守卫。
**撞号 000011 在同一 PR 中显式让出**:将 ingestion-engine 的 `000011_bmc_field_authorities`
重命名为 `000052_bmc_field_authorities`(最小代价),并补写相应 down path。

> 重命名的安全性论证:
>
> - ingestion-engine 的 `schema_migrations` 当前若记录 `version=11`,重命名后 migrator
>   会报 "missing version 11"。解决方式:在重命名 migration 上线时先在 DB 里执行
>   `UPDATE schema_migrations SET version = 52 WHERE version = 11`(单行,可回滚),
>   然后再部署 binary。这个 update 放在 cutover 运维脚本里,不走 migration 目录。
> - 如果 ingestion-engine 尚未在生产部署过(检查 migration 文件落盘时间与线上版本),
>   可以直接重命名无需 DB fix。需上线前由 DBA 确认。

### 步骤 2 — 撞号检测脚本

新增 `/cmdb-platform/scripts/check-migration-collision.sh`:

```bash
#!/usr/bin/env bash
# check-migration-collision.sh
#
# 扫描两个 migrations 目录,若同一数字前缀在两边都存在或在同一目录里出现多次,
# 则以非零退出码打印冲突行。
#
# 用法:
#   ./scripts/check-migration-collision.sh                # 扫默认两个目录
#   ./scripts/check-migration-collision.sh --strict       # 同时校验 docs/MIGRATIONS.md 覆盖面

set -euo pipefail

repo_root="$(cd "$(dirname "$0")/.." && pwd)"
dirs=(
    "$repo_root/cmdb-core/db/migrations"
    "$repo_root/ingestion-engine/db/migrations"
)
registry="$repo_root/docs/MIGRATIONS.md"
strict=0
for a in "$@"; do
    case "$a" in
        --strict) strict=1 ;;
        -h|--help)
            sed -n '2,10p' "$0"
            exit 0
            ;;
        *) echo "unknown flag: $a" >&2; exit 2 ;;
    esac
done

# ---------------------------------------------------------------
# 1. 收集所有 (version, service, path) 三元组
# ---------------------------------------------------------------
tmp="$(mktemp)"
trap 'rm -f "$tmp"' EXIT

for d in "${dirs[@]}"; do
    service="$(basename "$(dirname "$(dirname "$d")")")"
    # 仅抓 UP 文件;每条 migration 有且只能有 up
    find "$d" -maxdepth 1 -type f -name '[0-9]*_*.up.sql' \
        | awk -F/ -v svc="$service" '{
            file = $NF
            # 提取前 6 位数字
            if (match(file, /^([0-9]{6})_/, m)) {
                printf "%s\t%s\t%s/%s\n", m[1], svc, svc, file
            }
        }' >> "$tmp"
done

# ---------------------------------------------------------------
# 2. 撞号检测:同 version 出现 >1 次即 fail
# ---------------------------------------------------------------
collisions="$(awk -F'\t' '{print $1}' "$tmp" | sort | uniq -d)"
if [[ -n "$collisions" ]]; then
    echo "MIGRATION NUMBER COLLISION DETECTED" >&2
    echo "=====================================" >&2
    while IFS= read -r v; do
        echo "version $v occupied by:" >&2
        grep -P "^$v\t" "$tmp" | awk -F'\t' '{ printf "  - %s (%s)\n", $3, $2 }' >&2
    done <<< "$collisions"
    echo >&2
    echo "Resolution: pick the next available number from docs/MIGRATIONS.md" >&2
    echo "            and rename the later-to-land file." >&2
    exit 1
fi

# ---------------------------------------------------------------
# 3. up/down 配对检测:每个 up 必须有同名 down
# ---------------------------------------------------------------
missing_down=""
while IFS=$'\t' read -r v svc path; do
    up="$repo_root/$path"
    down="${up%.up.sql}.down.sql"
    if [[ ! -f "$down" ]]; then
        missing_down+="$path"$'\n'
    fi
done < "$tmp"
if [[ -n "$missing_down" ]]; then
    echo "UP migration missing matching DOWN:" >&2
    echo "$missing_down" >&2
    exit 1
fi

# ---------------------------------------------------------------
# 4. Registry 覆盖(仅 strict)
# ---------------------------------------------------------------
if [[ "$strict" == "1" ]]; then
    if [[ ! -f "$registry" ]]; then
        echo "registry missing: $registry" >&2
        exit 1
    fi
    not_in_registry=""
    while IFS=$'\t' read -r v _ _; do
        if ! grep -qE "^\| ${v} \|" "$registry"; then
            not_in_registry+="$v"$'\n'
        fi
    done < "$tmp"
    if [[ -n "$not_in_registry" ]]; then
        echo "migrations present on disk but missing from registry:" >&2
        echo "$not_in_registry" >&2
        echo "edit docs/MIGRATIONS.md and add a row per missing number." >&2
        exit 1
    fi
fi

# ---------------------------------------------------------------
# 5. 输出概要
# ---------------------------------------------------------------
count="$(wc -l < "$tmp")"
echo "OK — $count migrations across ${#dirs[@]} services, no collisions, all pairs matched."
```

授权:`chmod +x scripts/check-migration-collision.sh`(由 commit 附带)。

### 步骤 3 — CI workflow

新增 `/cmdb-platform/.github/workflows/migration-check.yml`:

```yaml
name: Migration Number Check

on:
  push:
    branches: [master, main, 'feat/*']
  pull_request:
    branches: [master, main]
    paths:
      - 'cmdb-core/db/migrations/**'
      - 'ingestion-engine/db/migrations/**'
      - 'docs/MIGRATIONS.md'
      - 'scripts/check-migration-collision.sh'
      - '.github/workflows/migration-check.yml'

jobs:
  collision-check:
    name: Detect migration number collisions
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - uses: actions/checkout@v4

      - name: Ensure script is executable
        run: chmod +x scripts/check-migration-collision.sh

      - name: Scan for collisions and missing down pairs
        run: ./scripts/check-migration-collision.sh

      - name: Validate registry coverage
        run: ./scripts/check-migration-collision.sh --strict

      - name: Print migration inventory (debug)
        if: always()
        run: |
          echo "== cmdb-core =="
          ls -1 cmdb-core/db/migrations | sort
          echo
          echo "== ingestion-engine =="
          ls -1 ingestion-engine/db/migrations | sort
```

### 步骤 4 — 解决现存撞号 000011

按 "ingestion 让号至 000052" 方案:

```bash
# 在一次 PR 里:
git mv ingestion-engine/db/migrations/000011_bmc_field_authorities.up.sql   \
       ingestion-engine/db/migrations/000052_bmc_field_authorities.up.sql
git mv ingestion-engine/db/migrations/000011_bmc_field_authorities.down.sql \
       ingestion-engine/db/migrations/000052_bmc_field_authorities.down.sql

# 更新 MIGRATIONS.md(已在步骤 1 的模板里体现)
```

同 PR 必须包含**生产环境 cutover 脚本**(不走 migrations 目录):

```sql
-- ops/cutover/2026-04-20-ingestion-migration-rename.sql
-- 在 ingestion-engine 所指向的 database 上执行(不是 cmdb-core 的 DB)
-- 在新版 ingestion-engine 部署 **之前** 运行
BEGIN;
-- 确认确实只有 11 号存在,没有脏版本
SELECT * FROM schema_migrations;
UPDATE schema_migrations SET version = 52 WHERE version = 11;
-- 再次确认
SELECT * FROM schema_migrations;
COMMIT;
```

如果 ingestion-engine 尚未投产,跳过 cutover SQL,直接重命名即可。实施前由 DBA 确认。

## 回滚方案

### CI 守卫回滚

删除 `.github/workflows/migration-check.yml` 即可恢复旧行为。脚本和 registry 文件是无害的,
可留在仓库中继续作为参考文档。

### Registry 回滚

`git revert` 对应 commit。registry 是纯文档,无副作用。

### 撞号重命名回滚

如果 ingestion-engine 已在生产执行了 `UPDATE schema_migrations SET version = 52`,
并后续发现 app 层配置/二进制未同步跟进(非常不常见),回滚步骤:

```sql
BEGIN;
SELECT * FROM schema_migrations;  -- 确认 52 存在、11 不存在
UPDATE schema_migrations SET version = 11 WHERE version = 52;
COMMIT;
```

同时在 git 中 `git revert` 重命名 commit。只有当**旧版 ingestion binary** 仍在生产滚动
部署、期望看到 version=11 时才需要此回滚;否则新 binary 会校验 version=52,不需回滚。

### 已撞号数据冲突

不适用:本次治理不改任何 `.up.sql` 内容,只改文件名和 registry。不会产生数据冲突。

## Commit 策略

按顺序:

1. `docs(migrations): add unified migration number registry`
   — 仅新增 `docs/MIGRATIONS.md`,不改其它
2. `chore(scripts): add migration collision detection script`
   — 仅新增 `scripts/check-migration-collision.sh` + `chmod +x`
3. `fix(ingestion): rename 000011 to 000052 to resolve collision with cmdb-core`
   — `git mv` + 更新 registry 对应行 + 附 `ops/cutover/` SQL
4. `ci(migrations): enforce migration number uniqueness in CI`
   — 新增 `.github/workflows/migration-check.yml`
5. `chore(make): expose check-migrations target`
   — Makefile 改动
6. `docs(codeowners): protect migration registry and guard scripts`
   — CODEOWNERS 改动(可并入 1 或单独)

**顺序重要**:先有 registry 和脚本(1+2),再解撞号(3),再上 CI(4)。否则 CI 一开就红,
阻塞别的 PR。
