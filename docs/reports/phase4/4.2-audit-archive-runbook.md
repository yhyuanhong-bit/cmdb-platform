# audit-archive Operations Runbook

> Companion to `/cmdb-platform/docs/reports/phase4/4.2-audit-monthly-partition-and-archival.md`.
> Audience: on-call engineers handling audit archive incidents.

## System at a glance

- `audit_events` is a monthly RANGE-partitioned table (migration
  `000053_audit_events_monthly_partitioning`).
- Partitions live as `audit_events_YYYY_MM`. One legacy catch-all
  (`audit_events_legacy_partition`) absorbs pre-partition history.
- The `audit-archive` CronJob runs at 02:00 UTC on the 1st of each
  month. It (1) creates the next-month partition if missing, and
  (2) drains the 13-month-old partition to S3 and DROPs it.
- Retention is enforced in two places: the shell wrapper's
  `-13 month` offset and the Go binary's `--retention-months` guard.
  Both must be bypassed simultaneously to lose hot data.

## Metrics and alert triggers

| Metric | Kind | Alert |
|---|---|---|
| `cmdb_audit_partition_count` | gauge | `< 3 for 1h` → WARN (next-month partition missing) |
| `cmdb_audit_archive_last_success_unix` | gauge | `time() - metric > 40*86400` → PAGE |
| `cmdb_audit_archive_failures_total` | counter, labelled by `stage` | `increase(...[2d]) >= 3` → PAGE |

The last two come from the CronJob via the node_exporter textfile
collector (`--metrics-path
/var/lib/node_exporter/textfile_collector/audit-archive.prom`).

## Common incidents

### A. `cmdb_audit_partition_count < 3`

Cause: the CronJob failed to pre-create next-month's partition.
Symptom window: usually starts the day after the monthly run.

1. Confirm the gauge by querying the DB directly:
   ```sql
   SELECT c.relname, pg_get_expr(c.relpartbound, c.oid)
   FROM pg_inherits i
   JOIN pg_class c ON c.oid = i.inhrelid
   JOIN pg_class p ON p.oid = i.inhparent
   WHERE p.relname = 'audit_events'
   ORDER BY c.relname;
   ```
2. Create the missing partition manually:
   ```sql
   BEGIN;
   CREATE TABLE IF NOT EXISTS audit_events_YYYY_MM
     PARTITION OF audit_events
     FOR VALUES FROM ('YYYY-MM-01') TO ('YYYY-MM+1-01');
   COMMIT;
   ```
3. Re-trigger the CronJob for this run's bookkeeping:
   ```bash
   kubectl -n cmdb create job --from=cronjob/audit-archive \
     audit-archive-manual-$(date +%s)
   ```
4. File a follow-up on why the previous run failed (look at
   `kubectl -n cmdb logs -l app.kubernetes.io/name=audit-archive`).

### B. `cmdb_audit_archive_failures_total{stage="detach"}`

Cause: `ALTER TABLE … DETACH PARTITION … CONCURRENTLY` failed. Common
reasons: open long-running transaction holding a lock on the
partition, replication lag on a logical replica, or pool exhaustion.

1. Find blocking sessions:
   ```sql
   SELECT pid, state, query_start, query
   FROM pg_stat_activity
   WHERE state != 'idle'
     AND query ILIKE '%audit_events%';
   ```
2. Verify no orphan `ALTER TABLE` is stuck in the `CONCURRENTLY` wait
   state (PG14 specifics):
   ```sql
   SELECT * FROM pg_stat_progress_cluster;
   ```
3. After the blocker is gone, re-run:
   ```bash
   kubectl -n cmdb create job --from=cronjob/audit-archive audit-archive-retry-$(date +%s)
   ```

### C. `cmdb_audit_archive_failures_total{stage="upload"}` or `stage="verify"`

Cause: the partition is DETACHED but the S3 upload or HEAD-verify
failed. The partition is an **orphan** — still in the DB but not
routed from writes. The Go binary logs this explicitly with
`partition orphaned; re-attach via runbook`.

1. Identify the orphan:
   ```sql
   SELECT c.relname
   FROM pg_class c
   LEFT JOIN pg_inherits i ON i.inhrelid = c.oid
   WHERE c.relname LIKE 'audit_events_____\___' ESCAPE '\'
     AND i.inhrelid IS NULL;
   ```
2. Decide: retry the upload, or re-attach?
   - **Retry upload** (preferred, keeps partition structure):
     ```bash
     # Don't call the CronJob — it would DETACH again and fail.
     # Run audit-archive directly, but skip DETACH by attaching
     # back first (idempotent step 3), then letting the CLI detach
     # cleanly on retry.
     ```
   - **Re-attach** (fastest unblock; use when S3 permanently
     unavailable):
     ```sql
     ALTER TABLE audit_events ATTACH PARTITION audit_events_YYYY_MM
       FOR VALUES FROM ('YYYY-MM-01') TO ('YYYY-MM+1-01');
     ```
   The partition becomes hot again; the retention window extends by
   one month. File a S3 incident ticket and re-run next cycle.

### D. `cmdb_audit_archive_failures_total{stage="drop"}`

Cause: upload succeeded (data is safe in S3) but DROP failed. The
table still exists but is fully detached. Low urgency.

1. Confirm the object is in S3:
   ```bash
   aws s3api head-object --bucket cmdb-audit-archive \
     --key audit/YYYY/MM/audit_events_YYYY_MM.parquet
   ```
   Check `ContentLength > 0` and `Metadata.sha256` present.
2. DROP manually:
   ```sql
   DROP TABLE audit_events_YYYY_MM;
   ```

### E. Write failing with "no partition found"

Cause: a write landed in a month with no pre-created partition.
This is only possible if scenario A went unaddressed for >24h.

Immediate mitigation: create the partition (see scenario A).
No data is lost — audit writes that failed are retried by the
caller (audit.Service.Record propagates the error).

## Manual operations

### Archive an arbitrary month (dry-run first)

```bash
kubectl -n cmdb run audit-archive-dryrun \
  --image ghcr.io/cmdb/audit-archive:latest \
  --rm -i --restart=Never \
  --env DATABASE_URL="$(kubectl get secret cmdb-db -o jsonpath='{.data.url}' | base64 -d)" \
  --env AWS_REGION=us-west-2 \
  -- --month 2025-03 --bucket cmdb-audit-archive \
     --retention-months 12 --dry-run
```

If the dry-run plan looks right, drop `--dry-run` and re-run.

### Restore a month back to the hot DB from S3

This is rare (subpoena, audit request). The parquet is written in
the same column order as `audit_events`; a duckdb one-liner rebuilds
an `audit_events`-shaped temp table that can be `INSERT ... SELECT`-ed
back. Coordinate with Security before doing this — restoring cold
data expands the hot-DB attack surface for compliance-sensitive rows.

```bash
# 1. Download the parquet
aws s3 cp s3://cmdb-audit-archive/audit/2024/04/audit_events_2024_04.parquet /tmp/

# 2. Load into a staging DB (NOT prod) and verify row count matches
#    the CronJob log's `rows` field for this month.
duckdb -c "SELECT count(*) FROM read_parquet('/tmp/audit_events_2024_04.parquet')"

# 3. If restore is approved, attach a new partition and INSERT.
#    This breaks the append-only invariant — get it signed off by
#    Security beforehand.
```

### Force next-month partition creation ahead of schedule

```sql
CREATE TABLE IF NOT EXISTS audit_events_YYYY_MM
  PARTITION OF audit_events
  FOR VALUES FROM ('YYYY-MM-01') TO ('YYYY-MM+1-01');
```

## Validation after migration 000053 rollout

```sql
-- 1. Parent + 5 children expected (legacy + current + next 3)
SELECT count(*)
FROM pg_inherits i
JOIN pg_class p ON p.oid = i.inhparent
WHERE p.relname = 'audit_events';

-- 2. Partition pruning is active
EXPLAIN SELECT * FROM audit_events
WHERE created_at >= date_trunc('month', now())
  AND tenant_id = '00000000-0000-0000-0000-000000000001';
-- expect: "Subplans Removed: N" in the Append node

-- 3. Append-only trigger still blocks DELETE
DELETE FROM audit_events WHERE id = gen_random_uuid();
-- expect: ERROR: audit_events table is append-only
```

## Legacy table cleanup

`audit_events_legacy` (the renamed pre-partition heap) is kept as a
safety net for 7 days after migration 000053 completes. Remove with
a follow-up migration once staging soak confirms nothing reads it:

```sql
DROP TABLE IF EXISTS audit_events_legacy;
```
