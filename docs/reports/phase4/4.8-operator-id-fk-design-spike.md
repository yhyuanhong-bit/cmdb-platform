# 4.8 `operator_id = uuid.Nil` 外键违反 —— 设计专题(Design Spike)

**Date:** 2026-04-20
**Commit at spike:** `eb72127` (master tip)
**Roadmap reference:** `REMEDIATION-ROADMAP.md` § 4.8 "`operator_id = uuid.Nil` 解决 FK 违反"
**Status:** Design spike — **no code changes in this document**. Implementation blocked on stakeholder decisions listed in § 12.

## TL;DR

Roadmap § 4.8 originally recommended inserting a global sentinel user with `id = uuid.Nil`.
After reading the schema, this recommendation does **not survive** Phase 1.3. Migration
`000050_users_username_tenant_scoped.up.sql` changed the uniqueness semantics to
`(tenant_id, username)` while `users.tenant_id` remains `NOT NULL REFERENCES tenants(id)`.
A single global sentinel cannot exist without picking **one** arbitrary tenant to own it,
which either (a) resurrects the cross-tenant `user` rows that Phase 1 just eliminated, or
(b) adds a nullable-tenant exception that conflicts with every existing tenant-scoping
invariant.

The spike therefore evaluates four concrete options, recommends **Option C** (keep FK,
add `operator_type` discriminator + NULL operator_id for system writes), and enumerates
every call site that must change. Option A2 (per-tenant sentinel) is the runner-up and
is viable if stakeholders insist on a non-NULL query column.

---

## 1. 目标

解决审计表(及同族 4 张表)当前已在生产环境潜在触发的外键违反问题,在多租户模型下为
"系统自动化"类写入提供一个**结构稳定、可查询、不泄漏跨租户身份**的 operator 语义。
具体目标:

1. 消除 `pgtype.UUID{Bytes: uuid.Nil, Valid: true}` 这种"写 all-zero UUID 到 FK 列"
   的反模式 —— Postgres 会在 FK 触发时直接 reject。
2. 保留审计链路对"谁做了这件事"的可解释性:UI / 报表 / 合规导出需要能区分
   - 真人用户(tenant-scoped, 可回溯到 `users` 行)
   - 系统自动化(scheduler, workflow, NATS subscriber, webhook 重试器)
   - 外部集成(`adapter_auto_disabled`, `integration_key_rotated`, sync envelope apply)
3. 不引入"一个全局 system 用户"这种穿透 tenant 边界的配置,避免 Phase 1 刚建立的
   `tenant_id` 结构性隔离被一条 INSERT 打回原形。
4. 迁移有确定性回滚路径,线上数据可 backfill。
5. 兼容 sqlc 已生成代码 —— 改动只落在 `internal/dbgen` 的重生成 + 调用点 helper,
   不破坏 `*.sql.go` 的幂等规则。

## 2. 现状调查

### 2.1 涉及的表与列

本 spike 范围内,`uuid.Nil` → FK 列 的危险模式至少出现在 5 张表:

| 表名 | 列 | FK 目标 | 迁移定义位置 | 可否 NULL |
|------|----|---------|--------------|----------|
| `audit_events` | `operator_id` | `users(id)` | `000008_audit.up.sql:8` | yes |
| `work_orders` | `requestor_id` | `users(id)` | `000005_maintenance.up.sql:11` | yes |
| `work_orders` | `assignee_id` | `users(id)` | `000005_maintenance.up.sql:12` | yes |
| `work_order_logs` | `operator_id` | `users(id)` | `000005_maintenance.up.sql:34` | yes |
| `inventory_tasks` | `assigned_to` | `users(id)` | `000007_inventory.up.sql:11` | yes |
| `inventory_items` | `scanned_by` | `users(id)` | `000007_inventory.up.sql:26` | yes |

所有列都是 **NULLABLE** 且都有 **real FK** 到 `users(id)` —— 不是 `DEFERRABLE INITIALLY
DEFERRED`,不是 `NOT VALID`,也不是 `ON DELETE SET NULL` 之外的特殊模式。Postgres 默认
`MATCH SIMPLE`:只要写入的 UUID 不是 NULL,就必须在 `users` 里找得到对应行,否则 INSERT
fail。

唯一一张是例外的是 `work_orders` —— `requestor_id` FK 从 Phase 0 审计时就没做特别处理,
所以 `w.maintenanceSvc.Create(ctx, tenantID, uuid.Nil, ...)` 13 个后台调用点全部在依赖
FK 默认不检查 all-zero UUID(**但它会检查** —— 见下 2.5)。

### 2.2 `audit_events` 表结构原样摘录

```sql
-- db/migrations/000008_audit.up.sql
CREATE TABLE audit_events (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id   UUID        NOT NULL REFERENCES tenants(id),
    action      VARCHAR(50) NOT NULL,
    module      VARCHAR(30),
    target_type VARCHAR(30),
    target_id   UUID,
    operator_id UUID        REFERENCES users(id),
    diff        JSONB,
    source      VARCHAR(20) NOT NULL DEFAULT 'web',
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_audit_events_tenant_created ON audit_events(tenant_id, created_at DESC);
CREATE INDEX idx_audit_events_target         ON audit_events(target_type, target_id);
CREATE INDEX idx_audit_events_operator       ON audit_events(operator_id);
```

迁移 `000028_audit_and_constraints.up.sql` 之后,该表进一步变成 append-only(触发器禁止
DELETE/UPDATE)—— 这意味着**一旦写错,修复只能走补偿/归档**,不能 UPDATE 修。

### 2.3 `users` 表关键约束

```sql
-- db/migrations/000002_tenants_and_identity.up.sql 摘录
CREATE TABLE users (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id     UUID         NOT NULL REFERENCES tenants(id),
    username      VARCHAR(100) NOT NULL UNIQUE,             -- ← 被 000050 替换
    password_hash TEXT         NOT NULL,                    -- ← 不能 NULL
    status        VARCHAR(20)  NOT NULL DEFAULT 'active',
    source        VARCHAR(20)  NOT NULL DEFAULT 'local',
    ...
);
```

迁移 `000050_users_username_tenant_scoped.up.sql` 后:

```sql
ALTER TABLE users DROP CONSTRAINT IF EXISTS users_username_key;
CREATE UNIQUE INDEX users_tenant_username_unique ON users (tenant_id, username);
```

也就是说: Phase 1.3 完成后,**"每个 tenant 都可以有一个 `username='system'` 的用户",
不会触发唯一冲突**。但 `id` 仍然是 PK,只有一行可以持有 `uuid.Nil` 作为 PK —— 如果选
option A1 (全局 sentinel),只能有一行,那一行必须挂在某个具体 tenant 上,这条 tenant
就成了"藏了一个穿透所有 tenant 的帐号"的污染源。

| # | 文件 | 行 | 说明 |

#### Group-C: `uuid.Nil` 只用作"是系统吗"的分支判断,不落盘

```go
    // System operations bypass approval checks
    return nil

```

语义,不影响 FK。保留不动。

### 2.5 FK 是否真的在 reject

**是。** Postgres `MATCH SIMPLE` FK 的行为是:

- 如果列是 NULL → 跳过 FK check(这就是为什么 Group-B 能工作)
- 如果列非 NULL → 必须在父表找到匹配行,否则 `ERROR: insert or update on table
  "audit_events" violates foreign key constraint "audit_events_operator_id_fkey"`

`'00000000-0000-0000-0000-000000000000'::uuid` 是一个**合法 UUID 值**,不是 NULL,所以
Postgres 会去 `users` 表查找 → 找不到 → reject。

**为什么线上看起来还在工作?** 三个猜测,按概率排序:

1. Group-A 的多数路径触发概率低。例如 firmware outdated 检测、EOS 检测、capacity alert
   需要真实数据且 rollup 到告警阈值,测试环境下很少触发。
2. 相关代码的 error 被静默吞掉。`maintenanceSvc.Create` 的返回 err 在 auto_workorders
   里大多被降级成 `zap.L().Warn(...) + continue`(详见 Phase 3.1 "Swallowed errors")。
   **审计表的 FK 违反正在默默发生**,只是没有人在 dashboard 上看到它。
3. 现网用户还没有启用对应的后台扫描 workflow(例如 `checkShadowIT` 依赖
   `network_scan_results` 表有数据)。

**验证手段(不在本 spike 内执行,列出来给实施阶段)**:

```sql
-- 当 Phase 4.8 开始实施时,先跑一次诊断
SELECT date_trunc('day', now() - interval '7 days'), count(*)
FROM pg_stat_user_tables ust
WHERE relname IN ('audit_events','work_orders','work_order_logs','inventory_tasks','inventory_items')
ORDER BY 1 DESC;

-- 从 postgres log 提取 FK 违反
-- (log_min_error_statement = 'error' 时)
grep 'audit_events_operator_id_fkey' /var/log/postgresql/postgresql.log
```

### 2.6 现有 sentinel 用户调查

grep `'00000000-0000-0000-0000-000000000000'`、`'system'` 作为 username:

| 结果 | 出处 | 是否真的存在 |
|------|------|-------------|
| `cmd/server/main.go:241` (seed admin) | 种子的是 `b0000000-...`,username `admin` | 不是 sentinel |
| `db/seed/seed.sql` | 同上 | 不是 sentinel |
| `rotate-integration-secrets/main.go:400` | 写 `audit_events` 时用 `target_type='system'`,不是 `users` 表 | 无 sentinel |
| `backfill-integration-secrets/main.go:299` | 同上 | 无 sentinel |

**结论**: 当前**没有任何 seeded system 用户**。roadmap 里提的"推测在某个地方有一行全
零 UUID 的系统用户让 FK 通过"—— 经过 grep 确认**不存在**。生产出现 FK 违反只是没有被
监控到。

### 2.7 相关的相邻约束

审计表在 `000028_audit_and_constraints.up.sql` 加了 append-only trigger:

```sql
CREATE TRIGGER audit_events_no_delete BEFORE DELETE ON audit_events ...
CREATE TRIGGER audit_events_no_update BEFORE UPDATE ON audit_events ...
```

这两个 trigger 意味着:
- 无法 `UPDATE audit_events SET operator_id = NULL WHERE operator_id = '00000000-...'::uuid`
  做 backfill。必须 SUPERUSER 临时 `ALTER TRIGGER ... DISABLE`,或者选择新增一张
  shadow 表导过去(不推荐)。
- 回滚方案(§ 8)必须显式考虑这两个 trigger。

## 3. 威胁模型 / 风险场景

### 3.1 生产可复现的 crash 场景

| 场景 | 当前结果 | 期望 |
|------|----------|------|
| Firmware outdated scanner 命中真实资产 → `auto_workorders.go:321` | `work_orders` INSERT 报 FK 违反 → `maintenanceSvc.Create` 返回 error → workflow `continue`,指标 `auto_workorders_total{outcome="error"}` +1,**工单没有创建** | 工单创建成功,requestor 可识别为"系统" |
| emergency alert → `notifications.go:171` 创建工单 | 同上 —— 关键告警没产出工单 | 工单创建成功 |
| sync_conflict cleanup → 写 audit | `audit_events` INSERT 失败 → 审计缺口 | 审计成功,可区分是清理任务 |
| impl.go:199 匿名/未知 session 写 audit | `audit_events` INSERT 失败 → 任何跨请求或者 ctx-user 丢失的路径**审计都不会写**(log 里有 warning,但审计链断) | 审计成功,`operator_type='unknown'` |

**严重度:** HIGH —— 这是审计完整性问题 + 关键业务流程(emergency work order)静默失败。

### 3.2 跨租户数据污染风险

如果采用 Option A1 (全局 sentinel user with `id=uuid.Nil`):

- 这个 sentinel 必须挂在某一个 tenant 下(因为 `users.tenant_id NOT NULL`)。
- Tenant A 的 `audit_events.operator_id = uuid.Nil` + Tenant B 的 `audit_events.operator_id
  = uuid.Nil` 都 FK 指向**同一个 tenant (X) 的 sentinel user**。
- 如果 RBAC 逻辑未来演化成"user 可以读自己 tenant 下所有人产生的 audit_events" (例如某种
  admin 视图),一个 tenant X 的 admin 就能通过 join `audit_events op ON op.id =
  users.id` 跨 tenant 看到 B 的审计 —— **tenant 边界被打穿一个针眼**。
- 另外,如果 tenant X 被 soft-delete 或者 `users.tenant_id` 的 FK 启用
  `ON DELETE RESTRICT`(当前是 NO ACTION),连带 users 失效 → 所有租户的
  system-action FK 全部失败。

这是为什么 roadmap 在 § 4.8 末尾自己打了一个 TODO 的原因。

### 3.3 报表 / 合规视图上的歧义

当前 `/audit` UI 把 `operator_id = NULL` 显示成空字符串,把 `uuid.Nil` 显示成
`'00000000-...'` —— **两种"系统"用不同方式呈现**。合规导出(SOC2 / ISO 27001)要求
每条审计事件必须能回答"这是 user X 做的 vs 系统自动做的 vs 外部集成做的"—— 用当前
schema 无法区分后两者。

## 4. 方案对比

四个候选方案(roadmap 原提 A + B,扩成 A1 / A2 / B / C / D):

### 方案 A1:全局 sentinel user (roadmap 原 A)

```sql
-- 必须先挑一个 "宿主 tenant"
INSERT INTO users (id, tenant_id, username, display_name, email, password_hash, status, source)
VALUES ('00000000-0000-0000-0000-000000000000',
        (SELECT id FROM tenants WHERE slug='_system' LIMIT 1),
        'system', 'System', 'system@cmdb.local', '!',
        'active', 'system');
```

需要新建一个"系统自用 tenant" `_system`,里面只装一个 system 用户。

**Pros:**
- `WHERE operator_id = '00000000-...'::uuid` 继续可查。
- 不需要 schema 改动(只需 seed + migration)。
- Go 侧改动最少,只要确保每个新 tenant 创建时也插入对应行。

**Cons:**
- 引入"系统 tenant"这个概念,和产品语义冲突:客户看到自己 tenant 列表里不应该有一个
  `_system`。
- RBAC 边界依赖应用层拦截`tenants WHERE slug='_system'`,而应用层一旦漏查就会泄漏
  (§ 3.2)。
- login / SSO / RBAC / 计费 /  quota / audit-export 多个代码路径都必须特判
  `tenant.slug = '_system'`,**技术债会扩散**。
- 无法回答 "operator_type" 问题:系统触发 vs 外部集成 vs 匿名 API 都挂同一个 user_id。

### 方案 A2:每租户一个 sentinel user(per-tenant system user)

```sql
-- 000051_seed_per_tenant_system_user.up.sql
INSERT INTO users (tenant_id, username, display_name, email, password_hash, status, source)
SELECT id, 'system', 'System', 'system@'||slug||'.local', '!', 'active', 'system'
FROM tenants
ON CONFLICT (tenant_id, username) DO NOTHING;

-- 以及 trigger: 新 tenant 自动 seed
CREATE OR REPLACE FUNCTION seed_system_user_for_new_tenant() RETURNS trigger AS $$
BEGIN
    INSERT INTO users (tenant_id, username, display_name, email, password_hash, status, source)
    VALUES (NEW.id, 'system', 'System', 'system@'||NEW.slug||'.local', '!', 'active', 'system')
    ON CONFLICT (tenant_id, username) DO NOTHING;
    RETURN NEW;
END $$ LANGUAGE plpgsql;
CREATE TRIGGER trg_tenant_seed_system_user AFTER INSERT ON tenants
FOR EACH ROW EXECUTE FUNCTION seed_system_user_for_new_tenant();
```

Go 端:新增 `identity.SystemUserID(ctx, tenantID) (uuid.UUID, error)`,对每个 tenant
查缓存 → 用这个 ID 替代 `uuid.Nil`。缓存是 `sync.Map` in-memory,1 小时 TTL(tenant 创建
后新 ID 基本不会变)。

**Pros:**
- FK 合法,每行都能 JOIN 回去。
- Tenant 隔离无缝 —— system user 的 `tenant_id` 就是它所在 tenant。
- 可查询(WHERE operator_id = :system_user_id),`users.source='system'` 过滤即可区分。
- 不需要"系统 tenant"的概念。

**Cons:**
- tenant 数 × 1 行 users,对当前 <100 tenant 量级无所谓,但对未来百万级 sharding 就是
  真实运维成本。
- 需要应用层维护 `systemUserID` 缓存。缓存失效时,每个审计写入多一次 DB 查 `system`
  用户 ID。
- 所有 login / 列表 / 导出代码必须过滤 `WHERE source != 'system'`(或对应策略),否则
  前端 user list 里会多出一行 `system` 用户。
- `password_hash` 必须是一个无法登录的值(`'!'` 或者随机 hash + lock 状态)。同时需要
  在 auth 路径拦截(`status = 'locked'` 或 check `source = 'system'` 直接 401)。

### 方案 B:`operator_id NULLABLE + 删除 FK`(roadmap 原 B)

```sql
-- 000051_audit_drop_operator_fk.up.sql
ALTER TABLE audit_events DROP CONSTRAINT audit_events_operator_id_fkey;
ALTER TABLE work_orders  DROP CONSTRAINT work_orders_requestor_id_fkey;
ALTER TABLE work_orders  DROP CONSTRAINT work_orders_assignee_id_fkey;
ALTER TABLE work_order_logs  DROP CONSTRAINT work_order_logs_operator_id_fkey;
ALTER TABLE inventory_tasks  DROP CONSTRAINT inventory_tasks_assigned_to_fkey;
ALTER TABLE inventory_items  DROP CONSTRAINT inventory_items_scanned_by_fkey;
```

应用层保证:`operator_id = uuid.Nil` 时,改成 NULL 下沉到 DB。

**Pros:**
- 实现最简单。
- 不需要 seed 数据。
- FK 已经在热路径(audit 写入)上增加了一次 index lookup,去掉对写性能有小改善
  (~50µs / insert,可忽略,但形式上更快)。

**Cons:**
- **损失引用完整性**。未来一个打字错误可能让 `operator_id` 指向已删除的用户 id 而不被
  检测到。审计是合规关键表,这是一个退步。
- 删除 FK 后,`users` 表的用户删除策略变得更微妙:原本 `ON DELETE NO ACTION` 会拦住
  "删除有 audit_events 的用户",删 FK 后就没人拦了。
- 无法在 `users` 和 `audit_events` 之间用 JOIN 做硬约束查询(应用层 join 仍然可行,但
  失去 DB 层保证)。

### 方案 C: 保留 FK + 新增 `operator_type` 判别列 + operator_id NULL 表示系统(推荐)

```sql
-- 000051_audit_operator_type.up.sql
CREATE TYPE audit_operator_type AS ENUM ('user', 'system', 'integration', 'sync', 'anonymous');

ALTER TABLE audit_events
    ADD COLUMN operator_type audit_operator_type NOT NULL DEFAULT 'user';

-- 约束:如果 operator_type = 'user', operator_id 必须非空且存在于 users(id);
-- 其他 type 下,operator_id 必须为 NULL。
ALTER TABLE audit_events ADD CONSTRAINT chk_audit_operator_type_id_match
CHECK (
    (operator_type = 'user' AND operator_id IS NOT NULL)
    OR (operator_type <> 'user' AND operator_id IS NULL)
);

-- 可选补充: 新建一个索引 (operator_type, created_at) 便于 "所有系统自动化行为" 查询
CREATE INDEX idx_audit_events_type_created ON audit_events(operator_type, created_at DESC);
```

类似的判别列加到 `work_order_logs`(`work_orders` 保持不变,语义上仍然需要一个 real
user 作为 requestor/assignee,"系统触发"这个场景下应该**挂到 tenant admin 名下** —— 见
§ 6.3)。

**Pros:**
- FK 保留,引用完整性 intact。
- 语义清晰:`operator_type` 直接回答"这是谁做的",UI / 合规 / 报表都有稳定字段。
- `WHERE operator_type='system'` 查询比 `WHERE operator_id IS NULL` 更表达意图。
- 不需要 seed 跨 tenant 数据。
- 未来扩展友好:加 `'webhook'`、`'scheduled_job'`、`'ai_agent'` 都只是 ENUM 扩展。
- 回滚成本低:drop 列即可退回原模式(旧代码读不到新列,没影响)。

**Cons:**
- 需要应用层的每个 audit 写入点根据来源显式设置 `operator_type`。这是一次性改动,但
  覆盖面大(17 处 Group-A)。
- 每行多 4 字节(PostgreSQL ENUM = oid, 4 bytes) + index,对百万级审计表新增 ~50MB
  存储。
- sqlc 需要重新 generate;测试要更新参数。

### 方案 D: 无 FK + 应用层 join tolerance(保留现状的理想化描述)

"假设现实就是 FK 已经不生效(DEFERRABLE 或被禁)"的路线。实测发现 FK **是**生效的,所以
D 等价于"刻意什么都不修"。列出来作为基线参照。

**Pros:** zero migration.
**Cons:** 问题继续存在,审计完整性继续漏。**不推荐。**

### 4.1 Trade-off matrix

| 维度 | A1 (global sentinel) | A2 (per-tenant sentinel) | B (drop FK) | C (operator_type + NULL) | D (do nothing) |
|------|----------------------|--------------------------|-------------|--------------------------|----------------|
| **查询 ergonomics** | `WHERE operator_id = uuid.Nil` 直接可用 | `JOIN users ON u.source='system'` | `WHERE operator_id IS NULL` | `WHERE operator_type='system'` (最清晰) | 现状 |
| **迁移成本** | 低(1 个 INSERT + 新 tenant trigger) | 中(每 tenant 一行 + trigger + 缓存) | 低(只 DROP FK) | 中(加列 + CHECK + 每个写入点更新) | 0 |
| **Cross-tenant safety** | **差**(§ 3.2) | 好 | 好 | 好 | N/A |
| **Referential integrity** | 完整 | 完整 | 丢失 | 完整 | 完整 |
| **Reporting clarity** | 差(system vs user 只靠 UUID) | 中(加 `users.source` 过滤) | 差(NULL 无语义) | **好**(显式 ENUM) | 差 |
| **RBAC complexity** | 高(`_system` tenant 每处特判) | 中(`system` user 每处过滤) | 低 | 低 | 低 |
| **审计/合规答题** | "操作者是 system user" | "操作者是当前 tenant 的 system" | "NULL" | "system"/"integration"/"user" | "NULL 或违规 UUID" |
| **扩展性**(更多 operator 类别) | 差 | 差(每增一类增一个 username) | 差 | **好**(ENUM 扩展) | 差 |
| **测试复杂度** | 中(需 seed 宿主 tenant) | 中(fixture 多一行) | 低 | 中(验证 CHECK 约束) | 低 |

## 5. 推荐方案 + 理由

### 推荐: 方案 C (operator_type 判别列 + NULL operator_id 表示系统),A2 作为 fallback

**核心理由:**

1. **语义清晰性是长期资产。** `operator_type` 用一次 ALTER TABLE 把"谁做的"的答案从
   "一个可能是 uuid.Nil 或真 UUID 的列"变成结构化字段,后续所有报表、UI、合规导出、
   跨 tenant 统计都受益。这是 A1/A2/B 都做不到的事。
2. **Tenant 边界 100% 不穿透。** 不需要宿主 tenant,不需要每 tenant 一行,不需要额外
   缓存。`operator_id IS NULL` 的语义是"无用户",在 FK MATCH SIMPLE 下自动绕过 FK 检查,
   DB 层完美自恰。
3. **引用完整性保留。** FK 不动。误代码产生的"指向已删除用户"bug 会继续被 Postgres
   拦住。
4. **扩展成本线性。** 以后想加 `'ai_agent'`、`'webhook'` 类别只要 `ALTER TYPE
   audit_operator_type ADD VALUE 'ai_agent';`,不用改别的。

**承认的损失:**

- 需要把 17 个 Group-A 调用点改成"显式选择 operator_type"。这是一个**必要**的改动 ——
  现在这 17 处都在 silently 用 uuid.Nil "表达" system 语义,但没有类型强制。迁移后
  从 IDE / linter 都能感知到这是三选一。
- `audit_events` 索引多一条,存储 +~50MB / 百万行。对一张本来就要分区归档的表(见
  roadmap § 4.2)这是可以接受的。
- 查询 `WHERE operator_id = uuid.Nil` 的 SQL 要改写 —— grep 全仓库,**没找到这种写法**
  (都是通过 ORM / sqlc,不存在裸 SQL),所以影响面为 0。

### A2 作为 fallback 的场景

如果 stakeholder 坚持"审计表每一行都必须 JOIN 回 users 拿到用户名展示,不接受 NULL",
那 A2 是 Plan B。实现模板见 § 6.5。A2 比 A1 好的一点就是 tenant 隔离不穿透,劣于 C 的
地方是报表层 `WHERE u.source='system'` 这种过滤得到处写。

## 6. 迁移步骤(以方案 C 为准)

### 6.1 Migration 000051 — audit 表判别列

```sql
-- db/migrations/000051_audit_operator_type.up.sql
BEGIN;

CREATE TYPE audit_operator_type AS ENUM (
    'user',         -- real user action via authenticated API
    'system',       -- scheduled workflows, background scanners
    'integration',  -- adapter pulls, webhook dispatcher, sync agent
    'sync',         -- cross-node sync apply (edge → central or reverse)
    'anonymous'     -- unauthenticated API (eg health-check triggered audit)
);

ALTER TABLE audit_events
    ADD COLUMN operator_type audit_operator_type NOT NULL DEFAULT 'user';

-- Backfill: rows where operator_id IS NULL cannot be 'user'; promote to 'system'
-- (pre-migration there was no NULL operator in production via happy paths; any
-- NULL is coming from metrics.go:300 — adapter auto-disabled — which is integration)
UPDATE audit_events SET operator_type = 'integration'
    WHERE operator_id IS NULL AND module = 'integration';
UPDATE audit_events SET operator_type = 'system'
    WHERE operator_id IS NULL AND operator_type = 'user';

-- Backfill: rows that historically landed uuid.Nil due to the bug we're fixing.
-- These slipped through because FK was actually rejecting them, but if any
-- environment was running with FK temporarily disabled, clean them up.
UPDATE audit_events SET operator_id = NULL, operator_type = 'system'
    WHERE operator_id = '00000000-0000-0000-0000-000000000000'::uuid;

-- CHECK:  user ⇒ id required; non-user ⇒ id must be NULL
ALTER TABLE audit_events ADD CONSTRAINT chk_audit_operator_type_id_match
    CHECK (
        (operator_type = 'user' AND operator_id IS NOT NULL)
        OR (operator_type <> 'user' AND operator_id IS NULL)
    );

CREATE INDEX idx_audit_events_type_created ON audit_events(operator_type, created_at DESC);

COMMIT;
```

**Note:** append-only trigger (`000028`) blocks UPDATE. The backfill UPDATEs above need
the trigger disabled inside the transaction:

```sql
-- prepended to the migration:
ALTER TABLE audit_events DISABLE TRIGGER audit_events_no_update;
-- ... UPDATEs ...
ALTER TABLE audit_events ENABLE TRIGGER audit_events_no_update;
```

`golang-migrate` does not run migrations as SUPERUSER by default; coordinate with DBA
or use `ALTER TABLE ... DISABLE TRIGGER USER` which the table owner can do.

### 6.2 Down migration

```sql
-- db/migrations/000051_audit_operator_type.down.sql

ALTER TABLE audit_events DROP CONSTRAINT IF EXISTS chk_audit_operator_type_id_match;
DROP INDEX IF EXISTS idx_audit_events_type_created;
ALTER TABLE audit_events DROP COLUMN IF EXISTS operator_type;
DROP TYPE IF EXISTS audit_operator_type;
```

### 6.3 `work_orders` 的处理(不同于 audit_events)

`work_orders.requestor_id` / `.assignee_id` 是业务关键字段,非审计。**不适合**用 NULL +
`operator_type` 方案:工单列表 UI 要渲染"申请人"列,"我的工单"视图依赖
`requestor_id = :current_user`。如果 system-triggered 的工单 requestor 是 NULL,这类工单
永远不会出现在任何"我的"视图里,也无 JOIN 可把名字补进列表。

**推荐做法:** system-triggered 工单的 `requestor_id` 挂到 per-tenant `system` 用户(mini
A2)。也就是说: **audit_events 走方案 C,work_orders / work_order_logs / inventory_* 族
走 mini A2。**按表语义裁剪,不是不一致。

Go 侧 helper:

```go
// internal/platform/identity/system_user.go (new file)
package identity

type SystemUserResolver struct {
    queries *dbgen.Queries
    cache   sync.Map // tenantID → cachedID{id, expiresAt}
    ttl     time.Duration // 1h default
}

// SystemUserID returns the UUID of the per-tenant `username='system'` user
// (seeded by migration 000052). Callers: workflow FK-safe inserts.
func (r *SystemUserResolver) SystemUserID(ctx context.Context, tenantID uuid.UUID) (uuid.UUID, error) {
    if v, ok := r.cache.Load(tenantID); ok {
        c := v.(cachedID)
        if time.Now().Before(c.expiresAt) {
            return c.id, nil
        }
    }
    u, err := r.queries.GetUserByTenantUsername(ctx, dbgen.GetUserByTenantUsernameParams{
        TenantID: tenantID, Username: "system",
    })
    if err != nil {
        return uuid.Nil, fmt.Errorf("resolve system user for tenant %s: %w", tenantID, err)
    }
    r.cache.Store(tenantID, cachedID{id: u.ID, expiresAt: time.Now().Add(r.ttl)})
    return u.ID, nil
}
```

### 6.4 Go 侧调用点迁移(Group-A 17 处)

#### audit_events 写入(通过 `audit.Service.Record`)

重签 `Record`,用 `OperatorType` enum(Go 常量)+ `*uuid.UUID`(nil 表示非 user):

```go
// internal/domain/audit/service.go
type OperatorType string
const (
    OperatorTypeUser        OperatorType = "user"
    OperatorTypeSystem      OperatorType = "system"
    OperatorTypeIntegration OperatorType = "integration"
    OperatorTypeSync        OperatorType = "sync"
    OperatorTypeAnonymous   OperatorType = "anonymous"
)

func (s *Service) Record(
    ctx context.Context, tenantID uuid.UUID,
    action, module, targetType string, targetID uuid.UUID,
    operatorType OperatorType, operatorID *uuid.UUID, // nil when operatorType != user
    diff map[string]any, source string,
) error {
    opID := pgtype.UUID{}
    if operatorType == OperatorTypeUser {
        if operatorID == nil {
            return fmt.Errorf("operator_type=user requires operator_id")
        }
        opID = pgtype.UUID{Bytes: *operatorID, Valid: true}
    } else if operatorID != nil {
        return fmt.Errorf("operator_type=%q must not carry operator_id", operatorType)
    }
    diffJSON, _ := json.Marshal(diff)
    _, err := s.queries.CreateAuditEvent(ctx, dbgen.CreateAuditEventParams{
        TenantID: tenantID, Action: action,
        Module:     pgtype.Text{String: module, Valid: true},
        TargetType: pgtype.Text{String: targetType, Valid: true},
        TargetID:   pgtype.UUID{Bytes: targetID, Valid: true},
        OperatorType: dbgen.AuditOperatorType(operatorType),
        OperatorID: opID, Diff: diffJSON, Source: source,
    })
    return err
}
```

Call-site updates for the 17 Group-A paths — pseudocode template:

```go
// BEFORE
s.auditSvc.Record(ctx, tenantID, "adapter_auto_disabled", "integration",
    "integration_adapter", adapterID, uuid.Nil, diff, "workflow")

// AFTER
s.auditSvc.Record(ctx, tenantID, "adapter_auto_disabled", "integration",
    "integration_adapter", adapterID,
    audit.OperatorTypeIntegration, nil /* no user */, diff, "workflow")
```

#### work_orders / work_order_logs 写入

Wire `SystemUserResolver` through workflow constructors:

```go
// BEFORE (auto_workorders.go:118)
order, err := w.maintenanceSvc.Create(ctx, tenantID, uuid.Nil, req)

// AFTER
systemUID, err := w.systemUsers.SystemUserID(ctx, tenantID)
if err != nil {
    zap.L().Warn("system user resolve failed", zap.Error(err))
    continue // skip this iteration; do NOT write with uuid.Nil
}
order, err := w.maintenanceSvc.Create(ctx, tenantID, systemUID, req)
```

#### tenant auto-seed trigger (per Phase 4.8 supporting migration)

Since work_orders still needs a real user, we also need A2's per-tenant system user
seed **only for the work-order family**. This is the same seed migration as the A2
fallback, but it's scoped narrower: we don't care about querying the `system` user
from audit table, we only need its `id` to use as a FK-safe requestor.

```sql
-- db/migrations/000052_per_tenant_system_user.up.sql
INSERT INTO users (tenant_id, username, display_name, email, password_hash, status, source)
SELECT id, 'system', 'System', 'system@'||slug||'.local', '!', 'active', 'system'
FROM tenants
ON CONFLICT (tenant_id, username) DO NOTHING;

CREATE OR REPLACE FUNCTION seed_system_user_for_new_tenant() RETURNS trigger AS $$
BEGIN
    INSERT INTO users (tenant_id, username, display_name, email, password_hash, status, source)
    VALUES (NEW.id, 'system', 'System', 'system@'||NEW.slug||'.local', '!', 'active', 'system')
    ON CONFLICT (tenant_id, username) DO NOTHING;
    RETURN NEW;
END $$ LANGUAGE plpgsql;

CREATE TRIGGER trg_tenant_seed_system_user
    AFTER INSERT ON tenants
    FOR EACH ROW EXECUTE FUNCTION seed_system_user_for_new_tenant();
```

#### Auth path guard

Make sure the `system` user cannot log in:

```go
// internal/domain/identity/auth_service.go — in Login(), right after fetching user
if user.Source == "system" {
    return nil, fmt.Errorf("authentication failed") // intentionally vague
}
```

And ensure `GET /users` list endpoints filter them out by default:

```sql
-- db/queries/users.sql: ListUsers
SELECT ... FROM users WHERE tenant_id = $1 AND source != 'system' ORDER BY ...
```

## 8. 回滚方案

### 8.1 Migration 层面

执行 `000051_audit_operator_type.down.sql`(§ 6.2)+ `000052_per_tenant_system_user.down.sql`:

```sql
-- 000052.down.sql
DROP TRIGGER IF EXISTS trg_tenant_seed_system_user ON tenants;
DROP FUNCTION IF EXISTS seed_system_user_for_new_tenant();
DELETE FROM users WHERE source = 'system';
```

**注意:** `DELETE FROM users` 会级联影响:
- `work_orders.requestor_id` 如果引用了 `system` 用户 ⇒ `ON DELETE NO ACTION` ⇒ 删除
  fail。
- 解决方法: rollback 前先 `UPDATE work_orders SET requestor_id = NULL WHERE requestor_id
  IN (SELECT id FROM users WHERE source='system')`。但这又要穿越 append-only trigger
  (work_orders 没 append-only,只 audit_events 有)—— 可行。

### 8.2 Go 代码层面

`git revert` 本 phase 对应的 commit。`dbgen` regenerate。注意: 如果迁移已经 down 掉
但代码仍然引用 `OperatorType` / `SystemUserResolver`,编译会 break。因此**回滚顺序必须
是 code first, schema second**。

### 8.3 应急 freeze 方案

如果生产出现新问题且无法等正式回滚,快速 mitigation:

```sql
-- 临时让 CHECK 放宽为 warning
ALTER TABLE audit_events DROP CONSTRAINT chk_audit_operator_type_id_match;
```

保留列不动,后续再重建 CHECK。

## 12. 开放问题 / 给 stakeholder 的问题

在实施前需要产品 + 架构 + 安全三方 sign-off 的点:

1. **[P0] 采纳方案 C 还是 A2?**
   C 更干净但改动更大(17 个调用点 + 签名变更)。A2 更保守(审计行 JOIN 到
   `system` 用户依然可 render user 名),但 `operator_type` 信息丢失,合规报表层需要
   额外过滤 `users.source`。**推荐 C,决定前不开工**。

2. **[P0] 方案 C 下 `audit_operator_type` ENUM 值的初始集合是否合理?**
   当前 5 值: `user / system / integration / sync / anonymous`。是否漏了?
   - `'ai_agent'` 留给 future,还是马上加?
   - `'webhook'` 是不是归 `integration`?
   - `'cli'`(`backfill-integration-secrets` 等)现在写的 source='admin-cli',要不要
     升格成 operator_type?

3. **[P1] work_orders.requestor_id 是否真的需要非空?**
   当前 schema 允许 NULL,但产品侧是否接受"申请人列显示 'System'"?如果接受,方案 C
   可以进一步推进到把 `work_orders.requestor_id` 也改成 NULL 语义 + 配套 UI 改造,
   migration 000052 就不需要了。

4. **[P1] 是否需要把"审计缺口"作为 SLO 指标?**
   当前 FK 违反 → `maintenanceSvc.Create` 返回 err → Workflow swallow → 没有指标。
   是否加 `audit_write_failures_total{reason,module}`?(这和 Phase 3.1 的
   swallowed-error 清理绑在一起更合理。)

5. **[P2] 历史 `audit_events` 中是否真有 `operator_id = '00000000-...'::uuid` 的行?**
   需要 DBA 在 staging + production 上各跑一次:
   ```sql
   SELECT count(*) FROM audit_events WHERE operator_id = '00000000-0000-0000-0000-000000000000'::uuid;
   ```
   如果 > 0,backfill 步骤(§ 6.1)必须保留并验证。如果 = 0,可以精简 migration。

6. **[P2] append-only trigger 在 migration 期间临时 disable 的风险窗口。**
   是否接受"migration tx 期间若有并发审计写入,trigger 暂时不生效"?如果不能接受,
   需要把迁移拆成两步:
   - Step A: add column with default, no backfill — 已 enforce append-only
   - Step B: 单独的 one-shot script(非 migration),停机窗口内跑 backfill

7. **[P2] Login 拒绝 `source='system'` 的错误信息是否泄漏存在性?**
   当前草案 `"authentication failed"` 和普通密码错误等价,不泄漏。确认安全团队接受。

8. **[P2] `systemUserID` 缓存失效策略。**
   1 小时 TTL 是否足够?或者改为"启动时加载所有 tenant → 缓存"(内存开销 = tenant 数 ×
   16 bytes,可忽略),然后订阅 `tenant.created` 事件做增量?—— 后者更简单也更确定,推荐。
