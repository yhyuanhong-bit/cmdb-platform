# Phase 4.9 — `publicPaths` / `resourceMap` 配置化(RBAC Config Externalization)

**Date:** 2026-04-20
**Roadmap reference:** `REMEDIATION-ROADMAP.md` § 4.9 "`publicPaths` / `resourceMap` 配置化"
**Service:** `cmdb-core/` (Go / Gin)
**Module:** `internal/middleware/rbac.go`
**Status:** Proposed

---

## 1. 目标

把 `cmdb-core/internal/middleware/rbac.go` 里两张硬编码的路由分类表
(`publicPaths`、`resourceMap`)从编译时常量挪到**运行时配置**,同时:

1. **上线新 endpoint 不需要重新发版才能配 RBAC**。新 API 只要走 `openapi.yaml`
   声明,RBAC 一侧只用改 `rbac.yaml` —— ops/security 团队也能 PR 审。
2. **跨环境一致性**:dev/staging/prod 的 `publicPaths` 完全一致 ——
   目前 `rbac.go` + `cmd/server/main.go` 两份 auth-bypass 列表就是 drift 温床。
3. **强 schema 校验**:启动时 fail-closed —— 未知 resource、拼写错的 action
   name、重复的 path 立即 fatal,而不是在请求到来时 default-deny(后者掩盖
   bug 直到客户投诉)。
4. **零性能回归**:RBAC middleware 在热路径(每个 API 请求都走一次);配
   置化后查询开销必须 ≤ 原 `map[string]bool` lookup。

> **非目标**:不引入数据库驱动的 RBAC(运行时编辑);不动 role / permissions
> schema;不改 middleware 签名(`func RBAC(queries, redisClient) gin.HandlerFunc`
> 保留)。

### 2.2 已知的 drift 源

`cmd/server/main.go:442-449` 在 middleware chain 里**再一次**硬编码了
auth bypass 列表(注意:auth bypass 和 RBAC public 不同,但在语义上重叠):

```go
// cmd/server/main.go:442-449
v1.Use(func(c *gin.Context) {
    path := c.Request.URL.Path
    if path == "/api/v1/auth/login" || path == "/api/v1/auth/refresh" || path == "/api/v1/ws" {
        c.Next()
        return
    }
    authMW(c)
})
```

- `main.go` 认为 `login / refresh / ws` 免 Auth;
- `rbac.go` 认为 `login / refresh / logout / ws` 免 RBAC。
- `logout` 在 `rbac.go` 里免 RBAC 但在 `main.go` 里**走 Auth**(必须拿着
  token 才能 logout,这是对的)。

这是"两份列表语义重合但不完全相同"的典型 drift 形态 —— 4.9 统一配置后,
`main.go` 的那段逻辑也应改为从同一份配置读(或至少引用 `middleware` 暴露
的常量)。

审计报告 `docs/reports/audit-2026-04-19/05-infra.md:78` 已经记下这条风险:

> `publicPaths` 硬编码(`rbac.go:22-28`)和 `cmd/server/main.go:341` 再
> 重复一次 auth bypass 列表。两份列表同步维护,已见过分叉风险。

### 2.3 已知的 resourceMap 语义错误

`docs/reports/audit-2026-04-19/05-infra.md:36` 记录了四处语义错标:

> `"bia": "system"`、`"quality": "system"`、`"discovery": "assets"`、
> `"upgrade-rules": "assets"` 让真正应当 scope 到 `bia` / `quality` 自己
> 资源的权限被按 `system` / `assets` 判定。

4.9 **不修**这些语义错误(那是 Phase 1 末尾或 Phase 2 的 security 事);
但配置化后修复一行 YAML 就能做,不必再发版。

### 2.4 历史 bug 样本

`docs/archive/page-audit-report.md:146-156` 记录:

> `notifications` 不在 `resourceMap` → `GET /notifications` 返回 403
> `unknown resource`;**Fix**:加 `"notifications": "system"` 到 `resourceMap`。

这是一个"新 endpoint 上线 → 忘记改 `rbac.go` → 前端黑屏"的典型事故。虽然
`default-deny` 在安全语义上是对的,但它把"配置漏洞"伪装成"权限不足",
提高了排障成本。4.9 之后这类漏配应在**启动时**就被 schema validation 捕捉
(见 §4.4 strict validation)。

### 2.5 当前读取路径(热路径)

```go
// rbac.go:84 — O(1) map lookup inside middleware
if publicPaths[c.Request.URL.Path] {
    c.Next()
    return
}

// rbac.go:153 — O(1) map lookup per request
return resourceMap[seg]
```

两次 `map[string]` lookup,每个请求一次。Benchmark 必须验证配置化后等价。

## 3. 风险 & 触发场景

### 3.1 部署新 endpoint 要重新编译发版

**场景**:产品决定加 `/api/v1/inventory/stocktake`(新 resource 类别),
上线流程当前是:

1. 改 `openapi.yaml` → 生成 handler stub。
2. 写业务代码。
3. **改 `rbac.go`** 把 `"inventory/stocktake"` 加到 map —— 容易忘。
4. 编译、二进制发布、滚动升级。

第 3 步漏掉的后果是步骤 1-2 看起来跑通了,但上线后第一个真实请求返回 403
`unknown resource`。需要重新走完整发版流程。

**发生过**:`notifications` 案例(上面 §2.4)就是这个模式。

### 3.2 多环境 RBAC 配置漂移

**场景**:staging 和 prod 的 roles 表从不同时间快照恢复,或一部分 role
permissions 在 prod 手动改过没回灌 staging。

当前 `publicPaths` / `resourceMap` 是编译常量,天然一致;但**权限内容**是 DB
里的 JSONB,容易漂移。4.9 之后 `rbac.yaml` 放到 git + ConfigMap,
drift 有 hash 可比对(见 §5.5 验证步骤)。

### 3.3 手指打错一个字母

**当前**:`"bia": "system"` 这种改错的字典值,在**编译时**过不了 review
也很难发现(都是字符串),要等有人真去请求 `/api/v1/bia/xxx` 且期望权限是
`bia:read` 而不是 `system:read` 才会出现权限错判。

**配置化后**:`rbac.yaml` 的 schema 可以白名单所有**允许的 resource 名称**,
在启动时校验 `resourceMap.values()` 必须是 `known_resources` 的子集 —— 否则
fatal。这比 Go 编译器强得多。

### 3.4 触发矩阵

| 触发动作 | 当前影响 | 4.9 后影响 |
|---------|---------|-----------|
| 上线新 URL segment | 需要改 Go + 编译 + 发版 | 改 rbac.yaml + ConfigMap reload |
| 笔误 `resource` 名 | 运行时才发现(可能被客户触发) | 启动时 fatal |
| staging/prod 配置漂移 | 无检测机制 | `cksum rbac.yaml` 纳入 release smoke |
| 列表修正(如 `bia: system → bia: bia`) | 改 Go 发版 | PR 改一行 + reload |
| 紧急回滚一条规则 | Git revert + 发版(≥30 min) | ConfigMap rollback + SIGHUP(秒级) |

## 4. 设计决策

### 4.1 配置源:YAML vs ConfigMap vs DB

| 选项 | 优点 | 缺点 |
|------|------|------|
| **YAML in git(默认)+ K8s ConfigMap 挂载** | 版本化、可 PR review、schema 可静态校验、dev 机可跑 | K8s-only;docker-compose 要 bind mount |
| 数据库驱动 + admin UI | 运行时改、按租户可覆写 | 引入"可编辑 RBAC 路由"这个新权限;错误配置 → 自锁管理员;每次请求多一次 DB 查询(即使缓存也要 invalidation) |
| etcd / Consul K/V | 真正的分布式配置 | 新组件 + 新故障面 |
| Env var(JSON blob) | 零新组件 | 5 KB YAML 塞 env 非常难读 |

**决策**:YAML in git。在 K8s 下通过 ConfigMap 挂载到 Pod `/etc/cmdb/rbac.yaml`;
在 dev / docker-compose 下通过 `CMDB_RBAC_CONFIG` env var 指向 repo 内路径
(fallback `./config/rbac.yaml`)。

> **不选 DB-driven 的关键原因**:RBAC 路由表是"安全基础设施",不该受
> admin UI 操作的影响。DB 驱动意味着"谁能写 roles 表就能改 routing" ——
> 与 4.9 的目标(让 ops/security 团队通过 git PR 参与 RBAC 管理)背道而驰。

### 4.2 Schema 校验方式

| 选项 | 优点 | 缺点 |
|------|------|------|
| **Go struct + `yaml.v3` + 启动时手写校验** | 零新依赖;可和业务 logic 紧耦合的自定义检查 | schema 描述散在代码里,不好给外部看 |
| JSON Schema + `github.com/xeipuuv/gojsonschema` | schema 是可被 IDE / CI 复用的独立文档 | 多一层依赖;YAML → JSON 转换有坑 |
| CUE | 类型最强、最多约束 | 学习成本;团队未用过 |

**决策**:Go struct + 手写校验。理由:
1. `cmdb-core/go.mod` 已经有 `yaml.v3`,无新依赖。
2. 自定义校验规则(如"public path 必须是 `/api/v1/*` 或 `/healthz|/metrics`")
   写成 Go function 最自然。
3. schema 的"文档化输出"另行生成(Go struct → markdown table),CI 里把
   生成物 diff 进 `docs/RBAC_SCHEMA.md`,团队外可读。

### 4.3 热重载:SIGHUP vs poll vs restart

| 选项 | 语义 | 决策 |
|------|------|------|
| 启动时读一次,不支持 reload | 改配置 = 重启 Pod | **一期选这个** — 符合现有"K8s ConfigMap 挂载 + Pod 滚动重启"流程,rollout 完整可追踪 |
| SIGHUP 触发 reload | 老 Pod 不重启,收 signal 后 atomic swap | 二期 — 有清晰需求再加,避免过早复杂化 |
| Poll `file.mtime` 每 10s | 改配置后 10 秒内生效 | 弃 — fs watcher 在 ConfigMap projection 下语义奇怪(symlink swap) |
| HTTP admin endpoint `POST /admin/rbac/reload` | 运行时触发 | 弃 — 又引入"谁能触发 reload" 的权限问题(自相矛盾) |

**决策**:4.9 一期**不做 hot reload**。配置是 ConfigMap,改配置 = 升级
ConfigMap → 触发 Deployment rollout(via checksum/config annotation)→
滚动重启 Pod。这是 K8s 原生玩法,审计友好。§5 仍给出 SIGHUP reload 的
**可选代码**,作为二期实施模板,但默认不启用。

### 4.4 校验规则(fail-closed)

启动时 `config.Validate()` 必须保证:

1. `publicPaths` 每条必须匹配 `^(/healthz|/metrics|/api/v1/.+)$`。
2. `publicPaths` 不允许 trailing slash(防止 `/foo` vs `/foo/` 语义歧义)。
3. `resourceMap` 的 **key** 必须是 URL-safe segment(`^[a-z0-9][a-z0-9-]*$`)。
4. `resourceMap` 的 **value** 必须是 `knownResources` 白名单成员(白名单
   和 `roles.permissions` JSONB 里使用的 resource 名称必须对齐 —— 这正是
   真正把 RBAC 配置和代码耦合起来的点;白名单在 Go 代码里常量化)。
5. 没有重复的 publicPaths 字符串。
6. `resourceMap` 的 key 不能同时出现在 `publicPaths` 的路径中(否则语义
   矛盾:一条路径到底是 public 还是需要 RBAC?)。
7. 配置文件不存在或不可读 → `zap.L().Fatal`(不 fallback 到 empty map,
   那会让**所有请求**走 `unknown resource` 路径被 deny —— 不是 fail-closed
   而是 fail-deadlock)。

白名单(Go 常量)示例:
```go
var knownResources = map[string]bool{
    "assets": true, "topology": true, "maintenance": true, "monitoring": true,
    "inventory": true, "audit": true, "dashboard": true, "identity": true,
    "prediction": true, "integration": true, "system": true, "sync": true,
    "bia": true, "quality": true,
}
```

这份白名单是**代码 + YAML 的契约**:新增 resource 既要在代码里登记
(因为 `roles.permissions` 的 key 也要跟着改),也要在 YAML 的 `resourceMap`
里映射。

### 4.5 环境变量 override

运维场景:staging 需要临时打开 `/api/v1/debug/dump` 作为 public(只在 staging)。
改 ConfigMap 太重,改 YAML 又会进 master。解法:

```
CMDB_RBAC_PUBLIC_PATHS_EXTRA=/api/v1/debug/dump,/api/v1/debug/config
```

逗号分隔的**追加**列表(不替换,防止误删)。同样支持:
- `CMDB_RBAC_RESOURCE_MAP_EXTRA=debug:system,internal:system`

env override 项走完全相同的 schema validation。生产环境应该**不设**这两个
env 变量(代码不 panic,只是 warn 一条"env override is non-empty in production
deploy mode")。

### 5.2 新增 config 结构体与加载器

**新文件**:`cmdb-core/internal/middleware/rbac_config.go`

```go
package middleware

import (
	"errors"
	"fmt"
	"os"
	"regexp"
	"strings"

	"gopkg.in/yaml.v3"
)

// RBACConfig is the parsed form of config/rbac.yaml. Fields are kept
// exported so tests in other packages can construct fixtures, but in
// production the only caller is LoadRBACConfig.
type RBACConfig struct {
	Version     int               `yaml:"version"`
	PublicPaths []string          `yaml:"publicPaths"`
	ResourceMap map[string]string `yaml:"resourceMap"`
}

// knownResources is the closed set of resource names that may appear as
// VALUES in ResourceMap and as keys in roles.permissions JSONB.
//
// Adding an entry here is a deliberate contract change — both the config
// YAML and the roles table need coordinated updates. Keep this list
// alphabetised so diffs are easy to review.
var knownResources = map[string]bool{
	"assets":      true,
	"audit":       true,
	"bia":         true,
	"dashboard":   true,
	"identity":    true,
	"integration": true,
	"inventory":   true,
	"maintenance": true,
	"monitoring": true,
	"prediction":  true,
	"quality":     true,
	"sync":        true,
	"system":      true,
	"topology":    true,
}

var (
	publicPathRe  = regexp.MustCompile(`^(/healthz|/metrics|/api/v1/.+)$`)
	segmentKeyRe  = regexp.MustCompile(`^[a-z0-9][a-z0-9-]*$`)
	defaultCfgEnv = "CMDB_RBAC_CONFIG"
	defaultCfg    = "config/rbac.yaml"
)

// LoadRBACConfig reads and validates the RBAC routing config.
//
// Precedence:
//   1. path parameter (non-empty wins)
//   2. $CMDB_RBAC_CONFIG env var
//   3. ./config/rbac.yaml
//
// After the file is loaded, env-var overrides are applied:
//   CMDB_RBAC_PUBLIC_PATHS_EXTRA   comma-separated, appended
//   CMDB_RBAC_RESOURCE_MAP_EXTRA   comma-separated "segment:resource" pairs
//
// Returns a validated config or an error that the caller MUST treat as
// fatal — do not start the HTTP server on a validation error.
func LoadRBACConfig(path string) (*RBACConfig, error) {
	if path == "" {
		path = os.Getenv(defaultCfgEnv)
	}
	if path == "" {
		path = defaultCfg
	}

	raw, err := os.ReadFile(path)
	if err != nil {
		return nil, fmt.Errorf("read rbac config %q: %w", path, err)
	}

	var cfg RBACConfig
	if err := yaml.Unmarshal(raw, &cfg); err != nil {
		return nil, fmt.Errorf("parse rbac config %q: %w", path, err)
	}

	applyEnvOverrides(&cfg)

	if err := cfg.Validate(); err != nil {
		return nil, fmt.Errorf("invalid rbac config %q: %w", path, err)
	}
	return &cfg, nil
}

// Validate enforces the schema described in docs/reports/phase4/4.9-*.md §4.4.
func (c *RBACConfig) Validate() error {
	if c.Version != 1 {
		return fmt.Errorf("unsupported rbac config version %d (expected 1)", c.Version)
	}

	seenPaths := make(map[string]bool, len(c.PublicPaths))
	for _, p := range c.PublicPaths {
		if !publicPathRe.MatchString(p) {
			return fmt.Errorf("publicPaths[%q]: path must match %s", p, publicPathRe)
		}
		if strings.HasSuffix(p, "/") && p != "/" {
			return fmt.Errorf("publicPaths[%q]: trailing slash not allowed", p)
		}
		if seenPaths[p] {
			return fmt.Errorf("publicPaths[%q]: duplicate entry", p)
		}
		seenPaths[p] = true
	}

	publicSegments := make(map[string]bool)
	for _, p := range c.PublicPaths {
		const prefix = "/api/v1/"
		if strings.HasPrefix(p, prefix) {
			rest := strings.TrimPrefix(p, prefix)
			seg := strings.SplitN(rest, "/", 2)[0]
			publicSegments[seg] = true
		}
	}

	if len(c.ResourceMap) == 0 {
		return errors.New("resourceMap must not be empty")
	}
	for k, v := range c.ResourceMap {
		if !segmentKeyRe.MatchString(k) {
			return fmt.Errorf("resourceMap key %q: must match %s", k, segmentKeyRe)
		}
		if !knownResources[v] {
			return fmt.Errorf("resourceMap[%q] = %q: unknown resource (add to knownResources if this is a new RBAC resource)", k, v)
		}
		if publicSegments[k] {
			return fmt.Errorf("resourceMap[%q]: segment is also listed in publicPaths — a path is either public OR RBAC-checked, not both", k)
		}
	}

	return nil
}

func applyEnvOverrides(c *RBACConfig) {
	if extra := os.Getenv("CMDB_RBAC_PUBLIC_PATHS_EXTRA"); extra != "" {
		for _, p := range splitCSV(extra) {
			c.PublicPaths = append(c.PublicPaths, p)
		}
	}
	if extra := os.Getenv("CMDB_RBAC_RESOURCE_MAP_EXTRA"); extra != "" {
		if c.ResourceMap == nil {
			c.ResourceMap = make(map[string]string)
		}
		for _, kv := range splitCSV(extra) {
			parts := strings.SplitN(kv, ":", 2)
			if len(parts) != 2 {
				// Validation below will also reject empty keys; this is
				// best-effort tolerant parsing.
				continue
			}
			c.ResourceMap[strings.TrimSpace(parts[0])] = strings.TrimSpace(parts[1])
		}
	}
}

func splitCSV(s string) []string {
	parts := strings.Split(s, ",")
	out := make([]string, 0, len(parts))
	for _, p := range parts {
		if trimmed := strings.TrimSpace(p); trimmed != "" {
			out = append(out, trimmed)
		}
	}
	return out
}
```

### 5.3 改 `rbac.go` 从配置读取

把 `rbac.go:25-62` 的两个 `var` 块**删除**,改 `RBAC(...)` 签名接收 config:

```go
// rbac.go — refactored signature
func RBAC(cfg *RBACConfig, queries *dbgen.Queries, redisClient *redis.Client) gin.HandlerFunc {
	if cfg == nil {
		panic("middleware.RBAC: config must not be nil")
	}
	// snapshot into local vars to avoid racing if we ever add hot-reload later
	publicPaths := cfg.PublicPaths
	publicIndex := make(map[string]struct{}, len(publicPaths))
	for _, p := range publicPaths {
		publicIndex[p] = struct{}{}
	}
	resourceMap := cfg.ResourceMap

	return func(c *gin.Context) {
		if _, ok := publicIndex[c.Request.URL.Path]; ok {
			c.Next()
			return
		}

		resource := extractResource(c.Request.URL.Path, resourceMap)
		if resource == "" {
			response.Forbidden(c, "access denied: unknown resource")
			c.Abort()
			return
		}
		// ... (rest unchanged)
	}
}

// extractResource now takes resourceMap as a parameter instead of reading
// package-level state.
func extractResource(path string, resourceMap map[string]string) string {
	const prefix = "/api/v1/"
	if !strings.HasPrefix(path, prefix) {
		return ""
	}
	rest := path[len(prefix):]
	seg := strings.SplitN(rest, "/", 2)[0]
	return resourceMap[seg]
}
```

### 5.5 (可选,二期)SIGHUP 热重载

如果未来决定支持 hot reload,在 `RBAC(...)` 里把 `publicIndex` 和
`resourceMap` 存到一个 `atomic.Pointer[loadedConfig]`:

```go
// OPTIONAL — not enabled in 4.9 phase 1
type loadedConfig struct {
    publicIndex map[string]struct{}
    resourceMap map[string]string
}

func RBAC(cfgPath string, queries *dbgen.Queries, redisClient *redis.Client) gin.HandlerFunc {
    var current atomic.Pointer[loadedConfig]

    loadAndSwap := func() error {
        cfg, err := LoadRBACConfig(cfgPath)
        if err != nil {
            return err
        }
        idx := make(map[string]struct{}, len(cfg.PublicPaths))
        for _, p := range cfg.PublicPaths {
            idx[p] = struct{}{}
        }
        current.Store(&loadedConfig{publicIndex: idx, resourceMap: cfg.ResourceMap})
        return nil
    }
    if err := loadAndSwap(); err != nil {
        panic(err) // caller (main) must catch via recover or pre-validate
    }

    // SIGHUP listener (one per process, guarded by sync.Once upstream)
    go func() {
        sigCh := make(chan os.Signal, 1)
        signal.Notify(sigCh, syscall.SIGHUP)
        for range sigCh {
            if err := loadAndSwap(); err != nil {
                zap.L().Error("rbac config reload failed, keeping previous", zap.Error(err))
                continue
            }
            zap.L().Info("rbac config reloaded")
        }
    }()

    return func(c *gin.Context) {
        lc := current.Load()
        if _, ok := lc.publicIndex[c.Request.URL.Path]; ok {
            c.Next()
            return
        }
        resource := extractResourceWith(c.Request.URL.Path, lc.resourceMap)
        // ...
    }
}
```

Atomic pointer swap 保证 mid-request 不会看到"半更新"的 map。如果 SIGHUP
reload 失败,保留旧配置,继续跑 —— fail-closed 的正确语义。

**决定:4.9 一期不启用这段**。理由见 §4.3。

## 7. 回滚方案

每一步都独立可 revert,且失败代价低(RBAC 是**启动时**生效,runtime 出问题
的窗口小):

1. **回到硬编码 maps**:`git revert <rbac_config.go 引入的 commit>` 一次性
   恢复 `rbac.go` 中的 `var publicPaths` / `var resourceMap`。代码一致,
   无 schema drift。
2. **保留 loader 但降级行为**:如果 `LoadRBACConfig` 在生产意外 fatal,
   可以通过临时 env `CMDB_RBAC_CONFIG_FAIL_OPEN=1` 改成 warn + fallback
   到 embedded defaults(不推荐,但作为"凌晨 3 点止血"的逃生舱)。
3. **ConfigMap rollback**:改错了 `rbac.yaml` 导致启动 fatal → `kubectl rollout
   undo deployment/cmdb-core` 回到上个版本的 ConfigMap + binary 组合。

`kubectl rollout undo` 的前提是 `checksum/rbac-config` annotation 把
ConfigMap 变更绑到 ReplicaSet 版本 —— §5.6 的 Helm/Kustomize 模板必须
**带上**这个 annotation,否则 ConfigMap 单独回滚不会触发 Pod 重启。

## 11. 开放问题 / TODO

### 11.1 运行时 API 编辑 RBAC:安全隐患评估

若将来需要 `POST /admin/rbac/reload` 或 `PUT /admin/rbac/config` 来支持
运行时改规则,要回答的问题:

1. **谁有权限改**?—— `system:write`?还是新增一个 `rbac:write` 避免 super
   admin 以外的人改?
2. **审计谁改了什么**?—— 需要落 `audit_events` 表;diff before/after YAML。
3. **防自锁**:如果 admin 把 `users` 移出 `resourceMap`,下一次他自己的
   请求就会被 default-deny。要么在 API 层强制保留一份"reserved resources"
   白名单,要么每次 reload 后对当前 caller 立即做一次 dry-run 权限测试。
4. **跨副本同步**:若 cmdb-core `replicas > 1`,API 改配置后必须通知所有
   副本 reload —— 走 Redis pub/sub 或 NATS?这和 §4.3 的 SIGHUP 讨论叠加。

**推荐**:4.9 一期**不做**。先落 YAML + ConfigMap,让 ops 团队通过 git PR
走审批流。3 个月后 review 是否真的有"热修 RBAC"的业务需求。

### 11.2 `knownResources` 的治理

`knownResources` 白名单既是 Go 常量,又和 `roles.permissions` JSONB 的 key
形成契约。目前没有**强制**两侧一致的机制(roles 表里可以写任何 key,
`mergePermissions` 只是 union,不 reject 未知 resource)。

**TODO**:加一条 DB 校验(trigger 或 CHECK constraint):`roles.permissions`
JSONB 的 keys 必须是 `knownResources` 的子集。这是 Phase 1/2 的遗留,不在
4.9 范围,但**4.9 让它变可能**(白名单现在有官方源头)。

### 11.3 Hot reload 的二期决策

§5.5 给了 SIGHUP reload 的可选代码。是否启用 depends on:

1. ops 团队是否真的经常改 `publicPaths`(观察 3 个月提交频率)。
2. 是否有"不想让 Pod 重启、不想断连的 WebSocket"的场景。
3. CPU 成本:`atomic.Pointer[loadedConfig]` 加一次 load,~2ns/op,可接受。

**建议**:3 个月后 review。若提交频率低(< 1 次/月),hot reload 纯属过度
工程;若高,再加 SIGHUP。

### 11.4 环境 override 的安全约束

`CMDB_RBAC_PUBLIC_PATHS_EXTRA` 目前是**追加**列表,即便在 production 也能
把新路径设为 public。如果 env 被误设,可能把敏感 endpoint(如
`/api/v1/integration/credentials`)暴露成 public。

**Mitigation 选项**:
- 选项 A:`CMDB_RBAC_PUBLIC_PATHS_EXTRA` 在 `deploy_mode=production` 下禁
  用(config loader 如果读到非空值 → fatal)。
- 选项 B:保留但启动时 log WARN + 把 extra 列表打 metrics(Prometheus 可以
  alerts 到 ops-oncall)。
- 选项 C:额外的白名单校验,只允许 `/api/v1/debug/*` 前缀被 env 追加。

**推荐**:选项 C —— 允许 debug 场景但封死 misuse 路径。

### 11.5 `logout` 的特殊地位

`/api/v1/auth/logout` 是唯一"RBAC-public 但 Auth-required"的路径 —— 见
§5.4 的显式例外。如果将来有第二条这样的路径(比如 `/api/v1/users/me`),
硬编码例外会重演当前 drift 问题。

### 11.6 `extractResource` 不支持嵌套 segment

当前 `extractResource` 只看 `/api/v1/{seg}/...` 的第一段。如果将来 API 演化

"monitoring" resource。但把 `resourceMap` 形态未来换成**前缀树**是个可预
见的演化方向。4.9 YAML schema `resourceMap: map[string]string` 先保持扁
平,二期升级到 `resourceMap: []Rule { prefix, resource }` 时平滑迁移。
