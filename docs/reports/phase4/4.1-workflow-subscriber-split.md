## 目标

把 `internal/domain/workflows` 包里的"神类" `WorkflowSubscriber`(单 struct、7 个 NATS subscriber + 9 个 `Start*` ticker 共挂在同一个 receiver 上)拆分为**按子系统独立 struct**的结构(notifications / autoworkorders / metrics / sla / cleanup / divergence / quality / webhook retention),引入一个 `workflows.StartAll(ctx, Deps)` 合成入口,让 `cmd/server/main.go` 从 9 次 `wfSub.StartXxx(ctx)` 调用收敛为一行。拆完之后每个子系统有**显式依赖、独立构造器、独立测试替身**;跨子系统共享的工具(`createNotification`、`opsAdminUserIDs`)下沉到 `workflows/internal/notify`(或同级 `shared.go`)成为纯函数或小接口,而不是"方法挂在神类上"。本次只做**结构重排 + 可观察性**,**不改任何业务规则**;现有 `subscriber.go:66-80` 的 7 条 `bus.Subscribe` 仍由拆分后的 subscriber struct 注册,NATS subject 名字、ticker 间隔、SQL 语义、metric 名称全部保持不变。

### 为什么"文件拆分 ≠ 解耦"—— 具体证据

1. **共享字段污染**:`auto_workorders.go:585` 的 `createShadowITWorkOrder` 调 `w.createNotification(...)`(定义在 `notifications.go:246`),两个"子系统"在编译单元上是**同一个 struct 的方法**,任何对 `WorkflowSubscriber` 字段的增删都会波及所有文件。
2. **构造器单入口**:`New(pool, queries, bus, maintenanceSvc, cipher)` 签名已 5 个参数,加一个子系统就加一个,新增 `WithQualityScanner` 是 functional option 规避扩张,但 qualityTenantListerOverride 这类**测试专用字段**已经泄漏到生产 struct(`subscriber.go:30-37`),证明神类约束已经在迫使实现作出妥协。
3. **main.go 串联 9 次 Start*****:一个实例上起 9 个 goroutine,context cancel 能传到每个 ticker 但关停顺序无法表达(例如"先停 metrics puller 再停 webhook retention")。
4. **测试替身膨胀**:`auto_workorders_test.go` / `quality_test.go` / `emergency_transition_test.go` 都要围绕完整 `WorkflowSubscriber` 构造,即使只测一个子系统 —— 这正是"interfaces defined at use site"这一 Go idiom 的反面。

## 风险 & 触发场景

### 如果不做 4.1

1. **耦合持续积累**:审计报告 `02-workflows.md:312` "但解耦不完整" 这一节已经明确,目前下游任何人要加一个 cron(例如 Phase 4.4 `ingestion-engine leader election`、Phase 2.12 `StartFailurePredictor`)都必须往 `WorkflowSubscriber` 加一个方法,并在 `main.go` 再加一行 `Start*`。字段会无限膨胀,`New(...)` 签名会不断 break。
2. **生产事故半径放大**:因为所有 goroutine 共享同一个 `*pgxpool.Pool` 持有者和同一个 `eventbus.Bus` 持有者,任何一个 ticker 里的死循环或 goroutine leak 都会拖累其他子系统的调度(实测:`divergence.go` 在启用状态下会用到 cipher,若 cipher 解密函数阻塞,其他 ticker 仍能跑但 struct-level 的 logger 调用链会混入 `divergence` 的 fields)。审计报告 `02-workflows.md:68-73` "关停路径是脏的" 仍未彻底修复 —— 虽然 `ctx` 已经正确地用 `signal.NotifyContext` 创建(`main.go:82-83`),但 goroutine 之间没有 `errgroup` 或关停顺序概念。
3. **测试脆弱**:目前 `quality_test.go:252` 行才能跑通 quality scanner 的单测,需要同时 stub `qualityTenantListerOverride`、`qualitySvc`、空 bus、真实 pgxpool mock。如果是独立 struct,相应单测应该在 ~50 行以内。
4. **新人上车成本高**:`subscriber.go` 变"门面文件",读完只能告诉新人"这里有 7 个订阅";但具体每个订阅处理什么、有没有幂等性要全文件跳,`grep -rn "func (w \*WorkflowSubscriber)"` 出来 40+ 行结果才能拼出全貌。

### 实际审计符号

- `02-workflows.md:49` "`alert.fired` 是字符串字面量" —— 拆分时顺手改为 `eventbus.SubjectAlertFired` 常量(见 4.1 本报告迁移步骤 Step 2)。
- `02-workflows.md:308` "拆分收益明确但运行时仍是一个对象" —— 正是本次工作要修复的。
- `02-workflows.md:313` "cross-文件隐式依赖 `opsAdminUserIDs` 被 5+ 调用" —— 拆分必须把这两个 helper 提到共享位置。

## 设计决策

### 选项 A:按子系统拆 struct,`StartAll` 做 composition(**推荐**)

```
internal/domain/workflows/
  subscriber.go                  // 只保留 Deps struct + StartAll 入口 (<60 行)
  notify/
    notifier.go                  // Notifier struct: createNotification + opsAdminUserIDs
    notifier_test.go
  subscribers/
    order_transitioned.go        // OrderTransitionedSubscriber
    alert_fired.go
    asset_created.go
    inventory_completed.go
    import_completed.go
    scan_differences.go          // SubjectScanDifferencesDetected + SubjectBMCDefaultPassword 合成
    register.go                  // RegisterAll(bus, deps)
  schedulers/
    sla.go                       // SLAScheduler struct + Start
    session_cleanup.go           // SessionCleanupScheduler
    conflict_discovery_cleanup.go
    webhook_retention.go
    metrics_puller.go
    warranty_checker.go
    asset_verification.go
    divergence.go
    quality_scanner.go
  auto_workorders/
    warranty.go, eol.go, lifespan.go, firmware.go,
    missing_assets.go, shadow_it.go, duplicate_serials.go, missing_location.go,
    scan_differences.go, bmc_security.go   // 每个 check* 独立文件
  adapters/
    registry.go, prometheus.go, zabbix.go, custom_rest.go, placeholder.go
  shared/
    semver.go, layers_util.go
```

- 优点:每个 struct 独立可替换,单测简单,`New*Scheduler(deps)` 小接口易 mock,`StartAll` 汇总为单入口。
- 缺点:**包边界变多**,short-term 编译器 import 重排成本较大;package-level private helper 要升级成 public 或走 internal 子包。工作量 ~7–8 天(见工作量估算)。

### 选项 B:保留单 `WorkflowSubscriber` struct,提取 Strategy 接口

只把 9 个 `Start*` 和 7 个 `on*` 提成 `WorkflowWorker interface { Start(ctx) }` 和 `Subscriber interface { Register(bus) }`,`WorkflowSubscriber` 变成**注册器**遍历 workers。struct 字段不变,但 `main.go` 从 9 次降为 1 次循环。

- 优点:改动小,~2 天。
- 缺点:没解决"单一 struct 持有 7 个字段被多文件共用"的核心耦合;只是把 `Start*` 语法糖化。审计报告结论 `02-workflows.md:312` "运行时仍是一个对象" 依然成立。

### 选项 C:把 subscriber 和 scheduler 分到两个完全独立的 package

`internal/domain/workflows/subscribers/` 和 `internal/domain/workflows/schedulers/` 各自维护自己的 struct,但**不再共享 `workflows.New`**。

- 优点:边界最清楚。
- 缺点:`createNotification` 和 `opsAdminUserIDs` 两个 helper 要放到第三个 package(比如 `workflows/notify/`),`auto_workorders/` 同时依赖两边,循环依赖风险要靠依赖倒置缓解;工作量 ~10 天。

### 选择:**选项 A**

- 审计报告 `02-workflows.md:320` 给出的具体建议就是"`workflows.StartAll(ctx, cfg)` + 独立 `NotificationSubscriber / AutoWorkOrderSubscriber / ScanReactionSubscriber`",与选项 A 一致。
- 选项 B 只是表层重构,不解决神类问题,审计 finding 会留着。
- 选项 C 收益递减:`createNotification` 和 `opsAdminUserIDs` 即使独立成 `notify` 包,也无需把 subscribers / schedulers 再分两个 package —— 它们的区别只是"事件驱动 vs 时间驱动",同源依赖(pool / queries / notifier)。

**唯一的风险点**:在 `Deps` struct 里如何避免再次退化为 5+ 字段神类。约束是 `Deps` **只在 StartAll 处出现**,每个 Subscriber / Scheduler 构造器**只拿它需要的字段子集**(例如 `NewSLAScheduler(pool, notifier)`,不接触 `cipher / bus`)。CI 加一条 lint 阻止 `Deps` 字段在非 `subscriber.go` 以外被直接引用。

```go
// Deps is the aggregated set of dependencies StartAll needs. Passing a
// single struct keeps main.go flat and makes it cheap to grow or shrink
// the workflow set without repeatedly re-threading parameters.
type Deps struct {
    Pool           *pgxpool.Pool
    Queries        *dbgen.Queries
    Bus            eventbus.Bus
    MaintenanceSvc *maintenance.Service
    Cipher         crypto.Cipher
    QualitySvc     qualityScanner
}

// StartAll is the single entry point main.go should call. It registers
// every NATS subscriber and spawns every scheduled goroutine. Each
// worker derives its lifecycle from ctx — the caller owns cancellation.
// Returns an error if a worker cannot be initialised (today: never, but
// future workers may require eager validation).
func StartAll(ctx context.Context, deps Deps) error {
    if deps.Bus == nil {
        zap.L().Info("workflows.StartAll: bus is nil, skipping registration")
        return nil
    }
    sub := New(deps.Pool, deps.Queries, deps.Bus, deps.MaintenanceSvc, deps.Cipher).
        WithQualityScanner(deps.QualitySvc)
    sub.Register()
    sub.StartSLAChecker(ctx)
    sub.StartSessionCleanup(ctx)
    sub.StartConflictAndDiscoveryCleanup(ctx)
    sub.StartMetricsPuller(ctx)
    sub.StartWarrantyChecker(ctx)
    sub.StartAssetVerificationChecker(ctx)
    sub.StartDivergenceChecker(ctx)
    sub.StartWebhookRetention(ctx)
    sub.StartQualityScanner(ctx)
    return nil
}
```

然后实现 `internal/domain/workflows/notify/notifier.go`:

```go
// Package notify exposes the small notification helper used by every
// workflow subscriber and scheduler. Extracted from workflows/notifications.go
// so auto_workorders, metrics puller, cleanup, and SLA can all depend on a
// tiny surface without pulling in the full workflow subscriber.
package notify

import (
    "context"
    "encoding/json"
    "fmt"

    "github.com/cmdb-platform/cmdb-core/internal/eventbus"
    "github.com/cmdb-platform/cmdb-core/internal/platform/telemetry"
    "github.com/google/uuid"
    "github.com/jackc/pgx/v5/pgxpool"
    "go.uber.org/zap"
)

const sourceNotifyCreate = "workflows.notify.create"

type Params struct {
    TenantID     uuid.UUID
    UserID       uuid.UUID
    Type         string
    Title        string
    Body         string
    ResourceType string
    ResourceID   uuid.UUID
}

type Notifier struct {
    pool *pgxpool.Pool
    bus  eventbus.Bus
}

func NewNotifier(pool *pgxpool.Pool, bus eventbus.Bus) *Notifier {
    return &Notifier{pool: pool, bus: bus}
}

func (n *Notifier) Create(ctx context.Context, p Params) error {
    if _, err := n.pool.Exec(ctx,
        "INSERT INTO notifications (tenant_id, user_id, type, title, body, resource_type, resource_id) VALUES ($1, $2, $3, $4, $5, $6, $7)",
        p.TenantID, p.UserID, p.Type, p.Title, p.Body, p.ResourceType, p.ResourceID,
    ); err != nil {
        return fmt.Errorf("insert notification: %w", err)
    }
    if n.bus == nil {
        return nil
    }
    payload, err := json.Marshal(map[string]string{
        "user_id": p.UserID.String(),
        "type":    p.Type,
        "title":   p.Title,
    })
    if err != nil {
        zap.L().Warn("notify.Create: marshal payload failed", zap.Error(err))
        telemetry.ErrorsSuppressedTotal.WithLabelValues(sourceNotifyCreate, telemetry.ReasonJSONUnmarshal).Inc()
        return nil
    }
```

```go
package schedulers

import (
    "context"
    "time"

    "github.com/cmdb-platform/cmdb-core/internal/domain/workflows/notify"
    "github.com/jackc/pgx/v5/pgxpool"
    "go.uber.org/zap"
)

const slaCheckInterval = 60 * time.Second

type SLAScheduler struct {
    pool     *pgxpool.Pool
    notifier *notify.Notifier
}

func NewSLAScheduler(pool *pgxpool.Pool, notifier *notify.Notifier) *SLAScheduler {
    return &SLAScheduler{pool: pool, notifier: notifier}
}

func (s *SLAScheduler) Start(ctx context.Context) {
    ticker := time.NewTicker(slaCheckInterval)
    go func() {
        defer ticker.Stop()
        for {
            select {
            case <-ctx.Done():
                return
            case <-ticker.C:
                s.checkBreaches(ctx)
                s.checkWarnings(ctx)
            }
        }
    }()
    zap.L().Info("SLA scheduler started", zap.Duration("interval", slaCheckInterval))
}

// checkBreaches and checkWarnings are the former WorkflowSubscriber
// methods, verbatim, with `w.pool` → `s.pool` and `w.createNotification`
// → `s.notifier.Create`. Full source of these two methods is moved from
// sla.go:70-161 and is not repeated here — this is a pure move.
```

3. **`auto_workorders` 的 10 个 checker 是否合理?**有人可能觉得 `WarrantyChecker` / `EOLChecker` / `LifespanChecker` 过度拆分(都是 24h 一起跑的姐妹操作)。**建议保留每 check 独立 struct 但共享同一个 `schedulers/daily_governance_scheduler.go`** 做 composition。评审点。
4. **测试数据库 fixture**:目前 `sla_test.go` 等使用 pgx testcontainers 或 dockertest;拆分后 test 数量 ×~2,CI 时间可能翻倍。**需要确认 CI 是否需要把 workflows 包的 test 跑在 `-short` 或 parallel worker 里**。与 DevEx 团队确认。
5. **`WorkflowSubscriber` 导出符号兼容性**:外部 consumer(如果存在)如果直接 `workflows.New(...)` 使用,将在 Step 8 被破坏。**需要 grep 全 repo + 下游所有 app** 确认没有外部引用,再执行 Step 8。初步 `grep -rn "workflows.New" --include="*.go"` 只命中 `cmd/server/main.go:581` 一处,Step 8 可直接进行。
6. **关停顺序是否要显式表达?**目前所有 scheduler / subscriber 平行 goroutine,不 care 谁先 exit。但如果后续 4.6 OTel trace 要求 span 在 shutdown 时有序 flush,需要引入 `errgroup.Group` 或 `workflows.Stopper` 接口。**本 phase 不做,但留 TODO**。
7. **审计 finding 跟踪**:`02-workflows.md` 列了 20+ 条 "当前风险或缺口"(checkShadowIT LIKE dedup、checkFirmwareOutdated lexicographical max、cleanup 吞错等)。这些**不在 4.1 范围**,但 4.1 拆分后应该让它们更容易独立修复。**建议 4.1 完成后,为每条开独立 issue,指派到 Phase 3/Phase 4 后续 sub-tasks**。
