## 目标

把 `GET /api/v1/dashboard/stats` 从目前 4 字段(`TotalAssets / TotalRacks / CriticalAlerts / ActiveOrders`)扩到 8+ 字段(新增 `EnergyCurrentKW / RackUtilizationPct / PendingWorkOrders / AvgQualityScore`);同时把**被动 60s TTL**替换为**事件驱动的主动失效**(domain event 触发时 Redis `DEL dashboard:stats:<tenant>`),使前端首屏既**信息充足**(不必再并发拉 `fleet-metrics / capacity-planning / energy / monitoring/alerts` 四个子接口拼齐 KPI),又**实时性强**(写入 → 下一次 stats 请求立刻拿到新数据,而不是等到满 60 秒)。本次工作**不改现有 4 字段的语义和数据源**,只是在它们旁边加字段 + 在 service 外围包一层 event subscriber;前端保持兼容的 response envelope。

```go
// internal/domain/dashboard/service.go:41-97
func (s *Service) GetStats(ctx context.Context, tenantID uuid.UUID) (*Stats, error) {
    cacheKey := fmt.Sprintf("dashboard:stats:%s", tenantID.String())

    // Try cache first (best-effort — skip on any error).
    if s.redis != nil {
        if val, err := s.redis.Get(ctx, cacheKey).Result(); err == nil {
            var cached Stats
            if json.Unmarshal([]byte(val), &cached) == nil {
                return &cached, nil
            }
        }
    }

    totalAssets, err := s.queries.CountAssets(ctx, dbgen.CountAssetsParams{TenantID: tenantID})
    // ... CountAlerts / CountWorkOrders / raw SQL count(*) racks ...

    stats := &Stats{TotalAssets: totalAssets, TotalRacks: totalRacks,
        CriticalAlerts: criticalAlerts, ActiveOrders: activeOrders}

    // Write-through cache (best-effort).
    if s.redis != nil {
        if data, err := json.Marshal(stats); err == nil {
            _ = s.redis.Set(ctx, cacheKey, string(data), statsCacheTTL).Err()
        }
    }
    return stats, nil
}
```

**关键观察**:

- `TotalAssets` / `CriticalAlerts` / `ActiveOrders` 走 sqlc (`CountAssets` / `CountAlerts` / `CountWorkOrders`),类型安全;
- `TotalRacks` 走 raw pgx `s.pool.QueryRow`(`service.go:78`),注释说明是历史遗留;
- 缓存 key 是 **per-tenant**,`dashboard:stats:<UUID>`;
- 写路径:**读 → 回写缓存**,TTL 60s。**没有任何 `Del` 调用点**,当前数据新鲜度完全靠 TTL 到期;
- `GetStats` 的 error 行为:任一 count 失败整体 500(handler `impl_audit.go:42-45`)。新增字段时**必须保证**一个字段查询失败不阻塞其余(见"设计决策")。

## 风险 & 触发场景

### 如果不扩字段(不做"扩充"半边)

1. **前端冗余请求**:`Dashboard.tsx` 已经挂了 3 个 TanStack Query,加载时三路并发(外加路由懒加载 + 图表组件),首屏 P90 耗时在 1.2s 左右(审计 `05-infra.md:203-205` 描述)。每个 KPI 卡片一个接口,典型 "N+1 endpoints on the UI side"。
2. **跨租户聚合一致性漂移**:rack_utilization 目前是在前端用 `useRackStats` 返回的 `total_slots / occupied_slots` 自己算百分比。后端如果不集中聚合,两个前端页面用不同公式就会出现同一 tenant "机架利用率 42% / 43.8%" 的不一致。
3. **缓存命中错失**:前端分别缓存 `rackStats / lifecycleStats / alerts`,但后端每个子接口各自走 DB。dashboard 本来是最适合做"合订缓存"的端点,一个 redis key 替代 4 个 handler 的冷启动。

### 如果不做主动失效(不做"失效"半边)

1. **最坏 60s 滞后**:用户创建资产后立即刷首页 → 资产数不变,直到 TTL 到期。客户在 demo 场景下容易踩(审计 `05-infra.md:199` "60 秒以内数据可能滞后")。
2. **critical alert 滞后更痛**:`CriticalAlerts` 是告警页最显眼的大字,firing → resolved 之后 60s 才更新。Ops 角色看这数字来判定"是否还有未处置告警",滞后会误导操作。
3. **Work order 完成状态**:工单从 `in_progress` 走到 `completed`,dashboard 的 `ActiveOrders` 60s 内不掉,首页红点仍显示 —— 与 maintenance 页自己的状态相矛盾。

### 真实审计符号

- `05-infra.md:197`:"Redis 缓存,key `dashboard:stats:{tenant_id}`,TTL 60 秒" + `:199` "**被动过期**,没有在 asset / alert / order 写路径做 invalidate。"
- `05-infra.md:205`:"dashboard 作为'聚合接口'实际上只聚合了一小半。"
- `05-infra.md:390`(整体评估第 12 条):"`dashboardSvc` 失效策略:在 asset/alert/order 写路径打一个 invalidate event,让 Redis stats 即时过期。"
- `05-infra.md:389`(整体评估第 11 条):"Dashboard 扩字段:目前只有 4 个计数,前端还要单独拉 5+ 个接口才能凑出首页,不如在 service 里扩 stats。"

## 设计决策

### 选项 A:在 service 里同步聚合所有新字段 + 订阅 `workflows.bus` 做失效(**推荐**)

```go
type Stats struct {
    TotalAssets         int64   `json:"total_assets"`
    TotalRacks          int64   `json:"total_racks"`
    CriticalAlerts      int64   `json:"critical_alerts"`
    ActiveOrders        int64   `json:"active_orders"`
    PendingWorkOrders   int64   `json:"pending_work_orders"`
    EnergyCurrentKW     float64 `json:"energy_current_kw"`
    RackUtilizationPct  float64 `json:"rack_utilization_pct"`
    AvgQualityScore     float64 `json:"avg_quality_score"`
}
```

- 扩 4 个字段在 `GetStats` 里顺序 select;每字段一条 sqlc query 或 raw SQL。
- 订阅 4 条 NATS subject(`SubjectAssetCreated / Updated / Deleted`、`SubjectAlertFired / Resolved`、`SubjectOrderTransitioned / Created`、`SubjectRackOccupancyChanged`)`DEL dashboard:stats:<tenant>`。
- 优点:**客户端接口单一、缓存语义清晰**(一个 key 管 tenant 的所有 dashboard KPI),主动失效和被动 TTL 双保险(即使失效事件丢了,60s 也兜得住)。
- 缺点:添加字段时 service 包需要依赖 `eventbus` 做 subscribe,把"读取服务"和"事件响应"混到一个 package。可接受,因为失效是 dashboard 的**运营责任**,不是其他域的责任。

### 选项 B:保持 4 字段,把新字段放独立的 `/dashboard/widgets/<name>`

- 每个 widget 独立 endpoint,独立缓存 key,独立失效。
- 优点:按需加载、单 widget 失败不影响其他。
- 缺点:前端要维护 N 个 hook,失败又回到现状。本 phase 的**目的之一**就是消灭这种碎片化。

### 选项 C:把失效从 Redis DEL 改成 pub/sub,前端也订阅

- 后端收到 domain event 后 `PUBLISH dashboard:invalidate <tenant>`,所有订阅的前端 / 其他进程触发 refetch。
- 优点:更实时,不依赖前端轮询。
- 缺点:**已经超纲**。前端已经有 WebSocket 订阅通知(通过 `workflows.notify.Create` + `SubjectNotificationCreated`),dashboard 可以复用同一条通道,但这是另一个 phase 的工作(实时推送)。本 phase 只做缓存失效,**不做推送**。

### 选择:**选项 A**

- 对齐 roadmap § 4.7 的字面描述:"扩字段 + 订阅事件 DEL Redis cache"。
- 扩展字段的用户价值最大,失效的运营价值最大,两件事一起做 review 一次。
- 跟 Phase 4.1 的 workflow 拆分**配合良好**:4.1 会把 subscriber 拆到 `workflows/subscribers/`,4.7 的 dashboard subscriber 可以按同一模式实现为 `dashboard.InvalidationSubscriber`。

### 如何防止"一个字段慢拖垮整个 stats"

新字段(特别是 `EnergyCurrentKW`)可能依赖 TimescaleDB,比普通 count 慢一个量级。需要:

1. **每字段独立超时**:`ctx, cancel := context.WithTimeout(ctx, 500*time.Millisecond)` 包 per-query call。超时返回 zero value + `dashboard_field_timeouts_total{field="..."}` counter。
2. **单字段失败不抛整体错误**:改 `GetStats` 签名返回 `(*Stats, []error)`,handler 层决定 -500 还是 -200 partial。
3. **文档化 partial 语义**:response 里加 `meta.partial: true`(延续 `response` envelope 的 meta 字段),前端可以灰显有问题的卡片。

### Step 4 — `RackUtilizationPct`(sum over rack_slots)

```sql
-- name: RackUtilizationByTenant :one
-- Returns (occupied_slots, total_slots) so the service can compute a
-- percentage and short-circuit when total_slots = 0. We deliberately
-- avoid dividing in SQL because Postgres returns NaN for 0/0 and we
-- don't want NaN in the JSON response.
SELECT
    COUNT(*) FILTER (WHERE asset_id IS NOT NULL)::bigint AS occupied_slots,
    COUNT(*)::bigint AS total_slots
FROM rack_slots rs
JOIN racks r ON r.id = rs.rack_id
WHERE r.tenant_id = $1 AND r.deleted_at IS NULL;
```

在 `service.go`:

```go
ru, err := s.queries.RackUtilizationByTenant(ctx, tenantID)

}
```

### Step 5 — `EnergyCurrentKW`(TimescaleDB metrics 聚合,最慢)

调研阶段:`metrics` 是 TimescaleDB hypertable(`000026_timescaledb_compression.up.sql`),命名约定 `metric_name = 'power.current_w'` 或 `'energy.draw_watts'`。需要产品确认 canonical metric name(**开放问题 #1**)。

假设命名确认为 `power.current_w`:

```sql
-- name: EnergyCurrentKWByTenant :one
-- Sum of the most recent power sample per asset, within the last 5
-- minutes. Timescale's time_bucket + first() returns the latest
-- observation for each asset efficiently. Returns watts; we divide by
-- 1000 in Go to get kW so the SQL is a pure aggregate.
SELECT COALESCE(SUM(latest_watts), 0)::double precision AS total_watts
FROM (
    SELECT asset_id, last(value, ts) AS latest_watts
    FROM metrics
    WHERE tenant_id = $1
      AND metric_name = 'power.current_w'
      AND ts >= now() - INTERVAL '5 minutes'
    GROUP BY asset_id
) latest;
```

在 `service.go`(注意强制 500ms 超时):

```go
energyCtx, cancel := context.WithTimeout(ctx, 500*time.Millisecond)
```

```go
// Invalidate drops the cached Stats for a tenant. Safe to call from any
// goroutine; uses the service's Redis client directly. A nil Redis
// client or a delete error is logged but does not propagate — the
// 60-second TTL is always a backstop.
func (s *Service) Invalidate(ctx context.Context, tenantID uuid.UUID) {
    if s.redis == nil {
        return
    }
    cacheKey := fmt.Sprintf("dashboard:stats:%s", tenantID.String())
    if err := s.redis.Del(ctx, cacheKey).Err(); err != nil {
        zap.L().Warn("dashboard: invalidate failed",
            zap.String("tenant_id", tenantID.String()),
            zap.Error(err))
        telemetry.DashboardInvalidateErrorsTotal.Inc()
        return
    }
    telemetry.DashboardInvalidateOkTotal.Inc()
}
```

```go
package dashboard

import (
    "context"
    "encoding/json"

    "github.com/cmdb-platform/cmdb-core/internal/eventbus"
    "github.com/google/uuid"
    "go.uber.org/zap"
)

// InvalidationSubscriber wires the Service.Invalidate call onto a set of
// domain events. Register it after the Service is constructed and once
// the event bus is ready. A nil bus makes this a no-op so tests and
// edge nodes without NATS still run.
type InvalidationSubscriber struct {
    svc *Service
}

func NewInvalidationSubscriber(svc *Service) *InvalidationSubscriber {
    return &InvalidationSubscriber{svc: svc}
}

// Register subscribes to every subject that should drop the stats
// cache. Order matters only in that we prefer over-invalidation to
// under-invalidation — it's cheap to recompute, expensive to show
// stale critical-alert counts.
func (s *InvalidationSubscriber) Register(bus eventbus.Bus) {
    if bus == nil || s.svc == nil {
        return
    }
    subjects := []string{
        eventbus.SubjectAssetCreated,
        eventbus.SubjectAssetUpdated,
        eventbus.SubjectAssetDeleted,
        eventbus.SubjectAssetStatusChanged,
        eventbus.SubjectRackCreated,
        eventbus.SubjectRackUpdated,
        eventbus.SubjectRackDeleted,
        eventbus.SubjectRackOccupancyChanged,
        eventbus.SubjectAlertFired,
        eventbus.SubjectAlertResolved,
        eventbus.SubjectOrderCreated,
        eventbus.SubjectOrderUpdated,
        eventbus.SubjectOrderTransitioned,
    }
    for _, subj := range subjects {
        bus.Subscribe(subj, s.handle)
    }
    zap.L().Info("dashboard invalidation subscribers registered",
        zap.Int("subjects", len(subjects)))
}

func (s *InvalidationSubscriber) handle(ctx context.Context, e eventbus.Event) error {
    tenantID, ok := parseTenantID(e)
    if !ok {
        zap.L().Debug("dashboard invalidate: event has no tenant_id",
            zap.String("subject", e.Subject))
        return nil
    }
    s.svc.Invalidate(ctx, tenantID)
    return nil
}

// parseTenantID extracts the tenant UUID from an event. Events carry
// tenant both as envelope field (e.TenantID) and sometimes inside the
// payload; we prefer envelope. Returns ok=false for system-wide
// events so we don't blow away the cache on truly global events.
func parseTenantID(e eventbus.Event) (uuid.UUID, bool) {
    if e.TenantID != "" {
        if id, err := uuid.Parse(e.TenantID); err == nil {
            return id, true
        }
    }
    // Fallback: look in payload for {"tenant_id": "..."}.
    var body map[string]json.RawMessage
    if err := json.Unmarshal(e.Payload, &body); err != nil {
        return uuid.Nil, false
    }
    raw, ok := body["tenant_id"]
    if !ok {
        return uuid.Nil, false
    }
    var s string
    if err := json.Unmarshal(raw, &s); err != nil {
        return uuid.Nil, false
    }
    id, err := uuid.Parse(s)
    return id, err == nil
}
```

## 开放问题 / TODO

1. **`energy_current_kw` 的 canonical metric name**:当前 `metrics` 表混合了 Prometheus(`node_*`)、Zabbix(自定义 key)、Custom REST(用户定义)三类,统一到 `power.current_w` 需要 **产品 + 集成团队确认**。如果选 `power.draw_watts` 或 `energy.current`,Step 5 的 SQL 需要调整。
2. **`avg_quality_score` 的口径**:目前假设是"每资产最新 score 的平均"。产品可能希望是"最近 24h 的 score 平均"(更抗离群),或"每资产 score 的中位数"。需要 **quality 团队确认**。
3. **`rack_utilization_pct`:是否按 U 数加权?**目前我给的 SQL 是 slot-based(一个 2U asset 占 2 个 slot 算 2)。如果 U 数非均匀(非 1U asset),这已经隐式正确;但如果希望按 rack U capacity 算(`SUM(slot_u) / SUM(rack.total_u)`),SQL 需要改。评审点。
4. **失效事件的粒度**:目前订阅 13 个 subject。`SubjectAssetUpdated` 在每次 asset CRUD 都发,可能是**过度失效**(一个 asset 改个 name,凭什么失效 stats?因为 status/serial 可能改)。可优化为只在 `SubjectAssetStatusChanged` 失效,但那要保证所有 status 变更都走 `status_changed` 事件(当前 `impl_assets.go` 不一定如此)。**先采取 over-invalidate 策略,观察 `dashboard_cache_hits_total / total_requests` 若 < 50% 再做精细化**。
5. **Multi-tenant 批量失效**:如果某次运维操作触发 100 tenant 同时有 event(例如 admin 统一 migrate status),会触发 100 次并发 Redis DEL。对 Redis 无压力(单 op),但对 subscriber goroutine 堆叠可能是问题。**需要确认 eventbus.Bus 的 handler 调度模型**(是一个 goroutine 还是 per-message goroutine)。NATS JetStream 是 per-message;如果 pool 里 handler 无锁共享 redis client 是安全的。暂不做限流。
6. **前端缓存策略**:TanStack Query 默认 5 分钟 stale time(per `useDashboardStats` 未设 `staleTime`)。**即使**后端失效立即生效,前端不重新 fetch,用户还是看到旧数 —— 除非用户切页触发 refetch。可搭配 WebSocket 推送,但超出本 phase。**现阶段**:把前端 `staleTime` 调低到 10s,或加 `refetchOnWindowFocus: true`,让后端失效和前端刷新协同。需要 frontend 团队决策。
7. **response envelope 里加 `meta.partial: true`**:当 Step 6 的某个字段降级为 0,response 是否要标记?对前端 UX 友好,但需要改 `response` 包和 `DashboardStats` 类型。**TODO**:先不加,观察是否有人抱怨"指标恒 0 但我没查到原因";若投诉多,下一个 sprint 加。
8. **Prometheus metric cardinality 审计**:`dashboard_field_timeouts_total{field=...}` 只有 8 个 label 值,安全。不引入高基数(如 `tenant_id`)label。
9. **Redis 故障演练**:手动停 Redis 看 `/dashboard/stats` 是否还工作(应该 fallback 到每请求 DB,性能下降但不 break)。加到 Chaos test runbook。
