# Phase 4.4 — ingestion-engine 多副本化(Horizontal Scaling)

**Date:** 2026-04-20
**Roadmap reference:** `REMEDIATION-ROADMAP.md` § 4.4 "ingestion-engine 多副本化"
**Service:** `ingestion-engine/` (Python 3.12 / FastAPI / Celery / asyncpg)
**Broker:** Redis (already shared with cmdb-core)
**Status:** Proposed

---

## 1. 目标

让 `ingestion-engine` 在 Kubernetes 下以 `replicas >= 2` 稳定运行,以满足:

1. **可用性**:任一副本(Pod)崩溃或滚动升级时,导入 / 发现 / MAC 扫描任务继续推进,不丢任务,不重复对同一台设备发起 SNMP 请求。
2. **水平扩展**:大批量 Excel 导入(10 万行级)可以并行跑在多个 Celery worker 上,吞吐量随副本数近似线性增长。
3. **无 split-brain**:需要"全局单例"的周期性作业(当前唯一例子:MAC/CDP 扫描)在任意时刻只由一个副本执行,且持有者崩溃后秒级切换。
4. **可观测性**:lock 的持有者、心跳、切主次数全部有 Prometheus metric + 结构化日志。

> **非目标**:本阶段不引入 etcd / Consul;不做跨 region 多活;不改 NATS 拓扑。所有协调原语都基于已有的 Redis 实例。

## 2. 现状

### 2.1 进程内周期循环(关键 split-brain 源)

`ingestion-engine/app/main.py:23-40` 在 FastAPI lifespan 内用 `asyncio.create_task`
起了一个 **while-True 循环**,每 5 分钟跑一次 MAC/CDP 扫描:

```python
# app/main.py:20
MAC_SCAN_INTERVAL_SECONDS = 300  # 5 minutes

# app/main.py:23-40
async def _periodic_mac_scan(application: FastAPI) -> None:
    """Background loop that runs MAC table scanning every 5 minutes."""
    await asyncio.sleep(60)  # Wait 1 min after startup for services to stabilise
    while True:
        try:
            from app.tasks.mac_scan_task import run_mac_scan

            pool = application.state.db_pool
            nats_client = application.state.nats_client
            result = await run_mac_scan(pool, nats_client, settings.tenant_id)
            logger.info(
                "Periodic MAC scan completed: scanned_ips=%s entries=%s",
                result.get("scanned_ips", 0),
                result.get("entries", 0),
            )
        except Exception:
            logger.warning("Periodic MAC scan failed", exc_info=True)
        await asyncio.sleep(MAC_SCAN_INTERVAL_SECONDS)

# app/main.py:54
scan_task = asyncio.create_task(_periodic_mac_scan(application))
```

这个循环的状态**完全存在于进程内存**(只有 `await asyncio.sleep` 和 Python
全局解释器锁在做协调),一旦把 `replicas: 1` 改成 `replicas: 2`,两个 Pod 会
**同时**触发 `run_mac_scan`。更糟的是它没有任何 cross-pod 去重机制。

`_periodic_mac_scan` 里唯一的租户来源是 `settings.tenant_id`(`config.py:34`,
**硬编码**为 `a0000000-0000-0000-0000-000000000001`)。多副本跑时每个副本都会
扫"这个唯一租户"——工作量直接翻倍。

全局 grep 结果确认只有这一处 `while True` / `asyncio.create_task` 在跑后台循环:

```
$ grep -nE 'while True|asyncio\.create_task|schedule\.every' app/
app/main.py:25:    await asyncio.sleep(60)
app/main.py:26:    while True:
app/main.py:40:    await asyncio.sleep(MAC_SCAN_INTERVAL_SECONDS)
app/main.py:54:    scan_task = asyncio.create_task(_periodic_mac_scan(application))
```

`SNMPCollector` / `SSHCollector` / `IPMICollector` / 所有 pipeline 函数都是
**stateless** —— 参数从 DB / Redis 读,结果回写 DB + NATS,无模块级状态。

- `task_acks_late=True` 意味着 worker 拉起任务后不会立刻 ack;任务**成功**
  后才 ack。
- 但 `task_reject_on_worker_lost` 缺省为 `False`,所以如果 worker 在跑任务
  过程中被 OOMKiller 杀掉,Celery 会**认为任务已被 ack**(broker 视角下 worker

  但只有在 redelivery 到期前任务可能实际丢失。
- Redis broker 的 visibility_timeout 缺省是 **1 小时**,意味着大 import 任务
  跑到 55 分钟崩溃后,还要等 1 小时才会被第二个 worker 捡起。

## 3. 风险 & 触发场景

### 3.1 Split-brain:多副本同时跑 MAC 扫描

**场景**:运维把 `deployment.yaml` 的 `replicas` 从 1 调到 3。

**后果**(从严重到轻微):

1. **SNMP 设备被打挂**:交换机的 SNMP agent 通常单线程,3 个副本在同一秒
   对同一台接入交换机发 `collect_cdp_neighbors` + `collect_mac_table`,容易
   把老款 Cisco Catalyst 2960 打到 CPU 95% 甚至 snmpd 崩溃。
2. **NATS 消息重复**:三个副本都 publish `mac_table.updated`,同一条 MAC
   记录在 cmdb-core 上 upsert 三次,`updated_at` 抖动,审计流水膨胀。
3. **DB 连接风暴**:`run_mac_scan` 的 `scan_targets` 查询和 `assets` 查询
   在 `for ip in ips` 循环里每条 ip 都可能 open 一次连接(代码里用 `async with
   pool.acquire()`),三副本并行会瞬间吃掉 asyncpg 连接池(默认 max=10)。
4. **Redis 缓存抖动**:凭证解密的 KDF 缓存(`credentials/encryption.py`)被
   三副本同时命中 miss,重复计算 scrypt,CPU 尖刺。

**概率**:100%,只要副本数 > 1 就必发。

### 3.2 任务丢失:worker OOM 或滚动升级时

**场景**:大 Excel(20 MB / 15 万行)import 跑到第 8 分钟,Celery worker Pod
因内存超限被 kernel OOMKiller 杀掉。

**当前行为**(Phase 3.11 配置):
- Broker 认为任务已被 consume,worker 断连后 **最多等 1 小时**(Redis
  visibility_timeout 缺省)才 redeliver。
- 如果 `task_reject_on_worker_lost=False`(当前状态),worker-lost 场景下任务
  **可能直接丢失**(取决于 Kombu 版本:新版本 redeliver,旧版本按 ack 处理)。
- 在这 1 小时内,UI 上的 `import_jobs.status` 卡在 `processing`,用户看着
  "99% / 99%" 的进度条不会动。

**后果**:用户抱怨"大 import 永远卡住",运维只能手动 `DELETE FROM import_jobs
WHERE id = ...` 然后让用户重传。对客户信任度是硬伤。

### 3.3 滚动升级期的窗口

**场景**:Helm upgrade 或 kubectl rollout restart,按默认 RollingUpdate 策略
逐个替换 Pod。

**当前问题**:
- 老 Pod 里的 `_periodic_mac_scan` loop 被 SIGTERM 触发 `scan_task.cancel()`
  —— 此时如果正在跑一次扫描(长的可能 2-3 分钟),scan 会在 `await`
  边界被中断,**已经打出去的 SNMP request 不会清理**(socket 由 gc 收)。
- 新 Pod 起来立刻开始 `asyncio.sleep(60)` 然后执行,和老 Pod 的"正在被 cancel"
  状态无协调 —— 可能 60 秒内两个 Pod 同时在不同进度跑扫描。

### 3.4 NATS publisher 丢消息

`publish_event(nats_client, "mac_table.updated", ...)` 用默认 fire-and-forget
模式。老 Pod 被 cancel 时,已经 serialize 但还没 flush 的消息会丢。这不是 4.4
独有的问题,但副本越多受影响次数越多。

### 3.5 触发矩阵

| 触发动作 | 影响 | Severity | 4.4 是否覆盖 |
|---------|------|---------|------------|
| `replicas: 1 → 2` 横向扩容 | MAC 扫描重复执行 | HIGH | ✓ |
| `replicas: 1 → 2` 横向扩容 | Celery 任务分散到 worker,需要重试保证 | MED | ✓ |
| worker Pod OOM | import 任务卡 1 小时或丢失 | HIGH | ✓ |
| kubectl rollout restart | 短窗口 split-brain + 未 flush NATS | MED | ✓ |
| Redis 主从切换 | 锁意外丢失(Redlock 才有这问题) | LOW | ✓(选型规避) |
| `localhost:8080` 不通 | discovery → staging 整条链路挂 | HIGH | ✓ |

## 4. 设计决策

### 4.1 Leader election 机制选型

| 选项 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| **Redis SETNX + EXPIRE**(单 Redis) | 已有 Redis;3 行 Lua 脚本搞定;心跳 renewal 简单;运维无新组件 | 单点故障时锁可能丢(Redis 挂导致所有 Pod 都认为自己是 leader——但此时 Redis 挂了,其他基础设施也废,业务本来就降级) | **选它** |
| Redlock(多 Redis 实例 quorum) | 可抵抗单 Redis 宕机 | 单 cluster 里没有多 Redis 实例;Martin Kleppmann 的批评(时钟偏移/GC pause 问题)意味着仍不是 100% 安全;运维复杂度飙升 | 弃 |
| etcd lease | 真正的 CP,时钟无关,自动 keepalive | 引入新组件,SRE 团队没运维过 etcd;开发机 / docker-compose 要额外起 etcd | 弃 |
| K8s `coordination.k8s.io/Lease` | 原生,控制平面保证;和 HPA 无缝 | 绑死 K8s(开发机 docker-compose 不支持);Python client 额外依赖;Pod 需要 RBAC 访问 lease API | 二期考虑 |

**决策**:单 Redis `SET NX PX` + Lua 续期脚本。接受"Redis 全挂时锁短暂丢失"
的风险——这个场景下 broker 也挂了,新扫描根本发不出去,即使 split-brain
也没有实际影响。

### 4.2 周期任务调度方式

| 选项 | 说明 | 结论 |
|------|------|------|
| **Celery beat(独立 Pod,replicas=1)** | Celery 原生支持;beat Pod 只 push 任务到 broker,不做扫描本身;worker Pod 横向扩;leader election 留给 beat pod 自身(用 K8s leader-elected Deployment 或 `celery-beat-redis-scheduler`) | **选它** |
| Celery beat embedded in worker | `celery worker -B` 启动,每个 worker 都跑 beat;多副本时多次触发 | 弃 |
| FastAPI 内 `asyncio.create_task` + Redis 锁 | 改动最小 | 混淆 HTTP 和后台 worker 的职责;FastAPI Pod 数量会影响扫描并发度(间接 split-brain 再次发生);不优雅 |

**决策**:Celery beat 作为**单独的 Deployment**(`replicas: 1`),挂一个
`celery-redbeat` scheduler(RedBeat 把 schedule 存 Redis,beat Pod 意外重启
不会丢 schedule state)。Beat 本身 `replicas=1` 没有 HA,但 beat 挂 2-3 分钟
完全无业务影响(最多晚一轮扫描)。如果真的需要 beat HA,未来升级到 K8s Lease
方案。

MAC 扫描 task 本身作为普通 Celery task,由多副本 worker 消费;task 内部再
持 Redis 锁防止 race(beat 只保证"每 5 分钟触发一次",但不保证"上一次已
结束"——如果扫描超过 5 分钟会叠加)。

### 4.3 锁粒度

**全局单例 vs per-tenant 单例?**

当前 `run_mac_scan` 收 `tenant_id` 参数,只扫该租户。今天 `settings.tenant_id`
硬编码为一个值,所以"全局 vs 每租户"效果相同。但 Phase 4 的下一步是多租户
ingestion(取消 `settings.tenant_id` 硬编码,改为从 `tenants` 表枚举),此时
锁必须是 **per-tenant**,否则 A 租户在跑时 B 租户会等。

**决策**:从 day-1 做 per-tenant 锁,key `ingestion:lock:mac_scan:{tenant_id}`。
当前单租户场景下行为等价,改多租户时零改动。

### 4.4 锁 TTL 与心跳

- TTL:**600 秒**(10 分钟),远大于扫描典型耗时(2-3 分钟),但小到 holder
  死掉后可接受的切主延迟。
- 心跳:每 60 秒续一次 TTL。用 Lua 脚本做 "CAS 续期"(只有 holder 是自己才
  延长),防止过期后被他人抢到再被自己续。
- 优雅退出:task 成功 / 异常结束时用 Lua 脚本 "CAS 删除"。

### 4.5 Config schema 迁移

`settings.tenant_id` 硬编码是 Phase 4 后续的问题(多租户 ingestion);4.4 暂不
动它,但把 leader election 的 key 写成 per-tenant 格式以备将来扩展。

## 5. 迁移步骤

### 5.1 新增 Redis 锁工具模块

**新文件**:`ingestion-engine/app/platform/redis_lock.py`

```python
"""Redis-based distributed lock with heartbeat renewal.

This implements the SET NX PX pattern plus a Lua-guarded release and renew
to make sure we only ever release or extend a lock that we still own. See
https://redis.io/docs/manual/patterns/distributed-locks/#correct-implementation-with-a-single-instance

Not suitable for transactional use (no Redlock, no fencing tokens). Good
enough for idempotent periodic work where the worst case of a dropped lock
is one duplicated scan after a Redis restart.
"""

from __future__ import annotations

import asyncio
import logging
import secrets
from contextlib import asynccontextmanager
from dataclasses import dataclass
from typing import AsyncIterator

import redis.asyncio as redis

logger = logging.getLogger(__name__)

# Release only if we still own the lock.
_RELEASE_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
"""

# Renew only if we still own the lock. Returns 1 on success, 0 otherwise.
_RENEW_SCRIPT = """
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
"""


@dataclass(frozen=True)
class LockAcquireResult:
    """Immutable result of an acquire attempt."""

    acquired: bool
    token: str | None
    holder: str | None  # current holder's token if not acquired; None if we got it


class LeaderLock:
    """Coordinates single-writer periodic work across replicas."""

    def __init__(
        self,
        client: redis.Redis,
        key: str,
        ttl_seconds: int = 600,
        heartbeat_seconds: int = 60,
    ) -> None:
        if heartbeat_seconds >= ttl_seconds:
            raise ValueError("heartbeat must be shorter than ttl")
        self._client = client
        self._key = key
        self._ttl_ms = ttl_seconds * 1000
        self._heartbeat_s = heartbeat_seconds

    async def try_acquire(self) -> LockAcquireResult:
        """Attempt to acquire the lock. Non-blocking."""
        token = secrets.token_hex(16)
        ok = await self._client.set(self._key, token, nx=True, px=self._ttl_ms)
        if ok:
            return LockAcquireResult(acquired=True, token=token, holder=None)
        current = await self._client.get(self._key)
        return LockAcquireResult(
            acquired=False,
            token=None,
            holder=current.decode() if isinstance(current, bytes) else current,
        )

    async def renew(self, token: str) -> bool:
        """Renew TTL if we still hold the lock. Returns True on success."""
        result = await self._client.eval(
            _RENEW_SCRIPT, 1, self._key, token, str(self._ttl_ms)
        )
        return bool(result)

    async def release(self, token: str) -> bool:
        """Release the lock if we still own it."""
        result = await self._client.eval(_RELEASE_SCRIPT, 1, self._key, token)
        return bool(result)

    @asynccontextmanager
    async def hold(self) -> AsyncIterator[str | None]:
        """Context manager: acquire, spawn heartbeat, release on exit.

        Yields the token string if acquired, None if not. Callers should check
        the yielded value and skip work when None.
        """
        outcome = await self.try_acquire()
        if not outcome.acquired:
            logger.info(
                "leader lock %s held by %s; skipping",
                self._key,
                outcome.holder,
            )
            yield None
            return

        token = outcome.token
        assert token is not None
        heartbeat = asyncio.create_task(self._heartbeat_loop(token))
        try:
            yield token
        finally:
            heartbeat.cancel()
            try:
                await heartbeat
            except asyncio.CancelledError:
                pass
            released = await self.release(token)
            if not released:
                logger.warning(
                    "lock %s was not ours at release time (lost leadership)", self._key
                )

    async def _heartbeat_loop(self, token: str) -> None:
        while True:
            await asyncio.sleep(self._heartbeat_s)
            ok = await self.renew(token)
            if not ok:
                logger.error(
                    "lost leader lock %s mid-work; stopping heartbeat", self._key
                )
                return
```

### 5.2 把 MAC 扫描从 FastAPI 循环搬到 Celery task

**删除** `app/main.py:20-67` 的 `_periodic_mac_scan` 与 `scan_task = asyncio.create_task(...)`。

**新文件**:`ingestion-engine/app/tasks/mac_scan_beat.py`

```python
"""Celery task that wraps mac_scan_task.run_mac_scan with leader election.

This is the task that celery beat enqueues every MAC_SCAN_INTERVAL_SECONDS.
Workers compete for a Redis lock scoped per tenant; only the winner runs
the scan. This is the only thing standing between multi-replica workers
and hammering the same switch three times in parallel.
"""

from __future__ import annotations

import asyncio
import logging

import asyncpg
import redis.asyncio as redis

from app.config import settings
from app.events import close_nats, connect_nats
from app.platform.redis_lock import LeaderLock
from app.tasks.celery_app import celery_app
from app.tasks.mac_scan_task import run_mac_scan

logger = logging.getLogger(__name__)

MAC_SCAN_LOCK_TTL_S = 600
MAC_SCAN_HEARTBEAT_S = 60


def _run_async(coro):
    loop = asyncio.new_event_loop()
    try:
        return loop.run_until_complete(coro)
    finally:
        loop.close()


@celery_app.task(
    bind=True,
    name="ingestion.periodic_mac_scan",
    # Infrastructure errors are retryable (short backoff since beat will
    # re-enqueue in ≤ 5 min anyway). Non-retryable errors fail the task
    # visibly so the beat metric catches the regression.
    autoretry_for=(asyncpg.PostgresError, OSError),
    retry_backoff=True,
    retry_backoff_max=120,
    max_retries=2,
)
def periodic_mac_scan_task(self, tenant_id: str) -> dict:
    """Entry point called by Celery beat."""
    return _run_async(_run(tenant_id))


async def _run(tenant_id: str) -> dict:
    redis_client = redis.from_url(settings.redis_url, decode_responses=False)
    lock = LeaderLock(
        client=redis_client,
        key=f"ingestion:lock:mac_scan:{tenant_id}",
        ttl_seconds=MAC_SCAN_LOCK_TTL_S,
        heartbeat_seconds=MAC_SCAN_HEARTBEAT_S,
    )
    try:
        async with lock.hold() as token:
            if token is None:
                # Someone else is scanning; not an error.
                return {"skipped": True, "reason": "leader_held_elsewhere"}

            pool = await asyncpg.create_pool(
                settings.database_url, min_size=2, max_size=10
            )
            nats = await connect_nats(settings.nats_url)
            try:
                result = await run_mac_scan(pool, nats, tenant_id)
                logger.info(
                    "mac scan completed tenant=%s scanned_ips=%s entries=%s",
                    tenant_id,
                    result.get("scanned_ips", 0),
                    result.get("entries", 0),
                )
                return {"skipped": False, **result}
            finally:
                await close_nats(nats)
                await pool.close()
    finally:
        await redis_client.aclose()
```

### 6.1 单元测试:lock 语义(fakeredis)

**新文件**:`ingestion-engine/tests/platform/test_redis_lock.py`

```python
import asyncio

import fakeredis.aioredis
import pytest

from app.platform.redis_lock import LeaderLock


@pytest.mark.asyncio
async def test_first_caller_acquires_second_skips():
    client = fakeredis.aioredis.FakeRedis()
    lock = LeaderLock(client, key="t", ttl_seconds=5, heartbeat_seconds=1)

    outcomes = []
    async def worker():
        async with lock.hold() as token:
            outcomes.append(token)
            await asyncio.sleep(0.1)

    await asyncio.gather(worker(), worker())
    # exactly one got a token, the other got None
    assert sum(1 for t in outcomes if t is not None) == 1
    assert sum(1 for t in outcomes if t is None) == 1


@pytest.mark.asyncio
async def test_heartbeat_keeps_lock_alive_past_ttl():
    client = fakeredis.aioredis.FakeRedis()
    lock = LeaderLock(client, key="t", ttl_seconds=2, heartbeat_seconds=1)

    async with lock.hold() as token:
        assert token is not None
        # wait longer than TTL; heartbeat should have renewed it
        await asyncio.sleep(2.5)
        current = await client.get("t")
        assert current.decode() == token


@pytest.mark.asyncio
async def test_release_does_not_affect_other_holder():
    client = fakeredis.aioredis.FakeRedis()
    lock_a = LeaderLock(client, key="t", ttl_seconds=60, heartbeat_seconds=30)
    lock_b = LeaderLock(client, key="t", ttl_seconds=60, heartbeat_seconds=30)

    r_a = await lock_a.try_acquire()
    assert r_a.acquired
    # simulate b trying with a stale token and attempting release
    released = await lock_b.release(token="not-our-token")
    assert released is False
    # a still holds it
    current = await client.get("t")
    assert current.decode() == r_a.token
```

Expected: `pytest -q` green; coverage for `redis_lock.py` ≥ 90%.

## 7. 回滚方案

每一步都独立可回滚:

1. **Revert beat + lock:** `git revert <beat-commit>` 回到 Phase 3.11 状态;
   把 `app/main.py` 的 `_periodic_mac_scan` 恢复 —— 但必须同时 `kubectl scale
   deployment ingestion-api --replicas=1`,否则又会 split-brain。

   和 beat schedule 段,保留 `task_reject_on_worker_lost` / `visibility_timeout`
   改动 —— 这两个本身是安全补丁。
3. **紧急止血(不改代码):** 把 `ingestion-worker` 缩回 `replicas: 1`,
   删掉 `ingestion-beat` Deployment,依赖 API Pod 里的进程内 loop 重新工作。
   这需要把 `main.py` 的 loop 通过 feature flag `ENABLE_LEGACY_MAC_LOOP=true`
   保留一段时间 —— 具体见"开放问题 TODO"。

**feature flag 落地建议**:

```python
# app/main.py
if settings.deploy_mode == "development" or os.environ.get("ENABLE_LEGACY_MAC_LOOP") == "1":
```

## 11. 开放问题 / TODO

### 11.1 service discovery 形式

`INGESTION_CMDB_CORE_URL` 应该写 K8s Service DNS(`http://cmdb-core.default.svc
.cluster.local:8080/api/v1`)还是经过 service mesh(Consul / Istio)?

- **option A**:K8s Service DNS。零依赖,但跨 namespace / 跨 cluster 时
  URL 形态不统一。
- **option B**:Consul catalog。统一抽象,但团队目前没运维 Consul。
- **option C**:Istio 内部 DNS + mTLS。和 option A 的 URL 一样,但多一
  层安全。

**建议**:目前 option A 够用,URL 通过 env 注入使得未来切 B/C 只需要改
values.yaml。此问题留到 Phase 4.6(OTel 全链路)再统一决策。

### 11.2 Celery beat HA?

单点 `replicas: 1` 的 beat Pod 如果 crash 2-3 分钟,最坏情况是错过一轮
MAC 扫描(5 分钟粒度)。对 CMDB 业务无实质影响,但有两个场景需要重新评估:

1. 未来 ingestion-engine 跑更高频任务(<1 分钟)时,beat 不可用窗口会造成
   数据新鲜度下降。
2. 多租户扩展后 beat 要 fan-out N 条 schedule,failure blast radius 变大。

**建议**:3 个月后 review。如果触发 A/B,升级到 K8s Lease-based beat
election(`celery-k8s-leader-election` 或自建)。

### 11.3 MAC 扫描锁丢失后如何告警

如果 `heartbeat` 意外失败(Redis 抖动 5 秒 > TTL)导致 `LeaderLock._heartbeat_loop`
记了 `lost leader lock ...`,当前只 log warn,不告警。

### 11.4 多租户 ingestion 时的锁粒度 vs 并发度

未来支持 N 个租户后,是否允许 N 个 MAC 扫描并发跑?

  in-flight 的扫描数。
