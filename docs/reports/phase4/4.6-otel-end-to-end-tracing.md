**Date:** 2026-04-20
**Source roadmap:** `docs/reports/audit-2026-04-19/REMEDIATION-ROADMAP.md` § 4.6
**Current commit at audit:** `eb72127` (master tip)
**Owner:** platform / observability squad
**Severity:** MEDIUM (operational — 影响故障定位 MTTR,非 exploitability 类问题)

---

## 目标

把目前**只有 Gin HTTP 入口**的半套 tracing 打通到 **pgx / NATS / Redis / 后台 workflow goroutine**, 达成:

1. **单次用户请求的完整 trace**: `HTTP POST /assets` → pgx queries → NATS publish →
   edge subscribe → 下游 pgx apply — 在 Jaeger / Tempo 中呈现为**一棵 span 树**,
   而非断裂的孤立 span。
2. **NATS 端到端上下文透传**: publisher 的 trace context 注入 `nats.Header`, subscriber
   提取后 continue span — "central publish envelope → edge apply envelope" 成父子链。
3. **后台循环** (`sync.reconciler`, `workflow.subscriber`, `monitoring.evaluator`,
   `quality.scanner`) 每轮 tick 作为 root span,名字稳定、按 span.name 可过滤。
4. **不放大采样**: 沿用 `ParentBased(TraceIDRatioBased(0.1))` — 父采样则子必采样,
   下游不会把 10% 放大成 100%。
5. **`OTEL_SDK_DISABLED=true` 一键关停**保留。

**反目标**: 不做 log / metric OTLP 输出(metric 留 Prometheus, log 留 zap);
不改 tracing backend 选型;span 不承载业务数据,只 semconv attrs。

## Trace 覆盖缺口

实际 `POST /api/v1/assets` 背后发生的调用链(读代码推导):

1. gin handler → `asset.service.Create`
2. pgxpool `INSERT INTO assets ...` (无 span)
3. pgxpool `SELECT ... FROM asset_events ...` (无 span)
4. `bus.Publish(SubjectAssetCreated, ...)` → NATS publish (无 span, 无 header 注入)
5. NATS 投递到 `sync.Service.onDomainEvent` — **断裂新 trace**
6. 查 sync_version, 构造 envelope, publish `CMDB_SYNC` (无 span)
7. Edge `sync.Agent.handleIncomingEnvelope` — 又一个**独立 trace**
8. 5-10 条 pgx UPSERT (无 span)
9. 写 sync_state (无 span)

Jaeger 里只看得见 step 1 本身。Step 5 的 subscribe 跟 step 1 的 HTTP 在不同 trace id
下 — MTTR 从"找一个 trace"变成"grep 日志按 correlation ID 拼凑"(而目前连统一
correlation ID 都没有)。

## 设计决策

### D1 pgx: `otelpgx` vs 手写 tracer

| Option                                | Pros                                               | Cons                                          |
|---------------------------------------|----------------------------------------------------|-----------------------------------------------|
| `github.com/exaring/otelpgx v0.10+`   | pgx v5 事实标准; `TraceQueryStart/End`; PII 脱敏开关 | 新增三方依赖                                   |
| 自写 `pgx.QueryTracer` 实现            | 零新依赖                                           | ~80 行 boilerplate; 连接级 span 绑定、error 传播全手写 |

**选 `otelpgx`**。pgx v5 README "Tracing" 小节明确推荐;
`WithIncludeQueryParameters(false)` 一键屏蔽 SQL 参数(避免 `password_hash` 进 span)。

### D2 NATS: 社区 tracer vs 手写 propagator

`go.opentelemetry.io/contrib/instrumentation/github.com/nats-io/nats.go/otelnats`
在 v0.67.0 contrib 下尚未 GA。场景极简(注/取 W3C traceparent 30 行代码),
**选手写** — 复用 `otel.GetTextMapPropagator()`(即 tracing.go 已配的 TraceContext),
一行不偏离标准。

### D3 Redis: `redisotel`

go-redis v9 官方包 `github.com/redis/go-redis/extra/redisotel/v9`:

```go
if err := redisotel.InstrumentTracing(client); err != nil { return nil, err }
```

**选 redisotel** — go-redis 维护方自家包, API 稳定。

### D4 Outbound HTTP: `otelhttp`

`go.opentelemetry.io/contrib/instrumentation/net/http/otelhttp`:

```go
httpClient := &http.Client{Transport: otelhttp.NewTransport(http.DefaultTransport)}
```

**选 `otelhttp`** — OTel 官方 contrib 核心包。

### D5 Propagator

维持 `TraceContext + Baggage`(tracing.go:51-54)。W3C 原生 — Grafana Tempo / Jaeger /
Datadog / Honeycomb 全支持。B3 是 Zipkin 遗留,无兼容负担。

### D6 采样

维持 `ParentBased(TraceIDRatioBased(0.1))`。后台 ticker 的 root span 也走这条,
每次 tick 10% 概率采样 — 符合期望。若未来"evaluator 每 tick 必采",再给
这类 span 挂 `AlwaysOnSampler`(follow-up, 见开放问题 #1)。

### D7 Span name 约定(遵循 OTel semconv v1.26+)

| Operation              | Span name                   | Kind      |
|------------------------|-----------------------------|-----------|
| Gin HTTP request       | `HTTP {method} {route}`     | SERVER    |
| pgx query              | `{op} {table}` (`SELECT assets`) | CLIENT |
| NATS publish           | `{subject} publish`         | PRODUCER  |
| NATS subscribe receive | `{subject} receive`         | CONSUMER  |
| Redis call             | `{command}` (`GET`,`SETEX`) | CLIENT    |
| Outbound HTTP          | `HTTP {method}`             | CLIENT    |
| Background ticker tick | `{ticker_name}.tick`        | INTERNAL  |

各 instrumentation 库按 semconv 默认生成,不手写。

func NewPool(ctx context.Context, databaseURL string) (*pgxpool.Pool, error) {

    if err != nil { return nil, fmt.Errorf("create pool: %w", err) }
    if err := pool.Ping(ctx); err != nil { return nil, fmt.Errorf("ping database: %w", err) }

### Step 4 — NATS publish 注入 `internal/eventbus/nats.go`

```go
import (
    "github.com/nats-io/nats.go"
    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/attribute"
    "go.opentelemetry.io/otel/codes"
    "go.opentelemetry.io/otel/trace"
    semconv "go.opentelemetry.io/otel/semconv/v1.26.0"
)

// natsHeaderCarrier adapts nats.Header to TextMapCarrier.
type natsHeaderCarrier nats.Header

func (c natsHeaderCarrier) Get(key string) string { return nats.Header(c).Get(key) }
func (c natsHeaderCarrier) Set(key, value string) { nats.Header(c).Set(key, value) }
func (c natsHeaderCarrier) Keys() []string {
    keys := make([]string, 0, len(c))
    for k := range c { keys = append(keys, k) }
    return keys
}

// Publish — Phase 4.6 包装 producer span 并注入 W3C traceparent.
func (b *NATSBus) Publish(ctx context.Context, event Event) error {
    subject := event.Subject
    if event.TenantID != "" { subject = subject + "." + event.TenantID }

    tracer := otel.Tracer("cmdb-core/eventbus")
    ctx, span := tracer.Start(ctx, subject+" publish",
        trace.WithSpanKind(trace.SpanKindProducer),
        trace.WithAttributes(
            semconv.MessagingSystem("nats"),
            semconv.MessagingDestinationName(subject),
            attribute.String("messaging.operation", "publish"),
            attribute.Int("messaging.message.payload_size_bytes", len(event.Payload)),
        ),
    )
    defer span.End()

    msg := &nats.Msg{Subject: subject, Data: event.Payload, Header: nats.Header{}}
    otel.GetTextMapPropagator().Inject(ctx, natsHeaderCarrier(msg.Header))

    if _, err := b.js.PublishMsg(ctx, msg); err != nil {
        span.RecordError(err)
        span.SetStatus(codes.Error, err.Error())
        return fmt.Errorf("publish %s: %w", subject, err)
    }
    return nil
}
```

**关键**: 原 `b.js.Publish(ctx, subject, payload)` 改 `b.js.PublishMsg(ctx, msg)` 以带 header。

### Step 5 — NATS subscribe 提取

```go
// Subscribe — Phase 4.6 从 header 提取 trace context, 起 CONSUMER span 挂到上游.
func (b *NATSBus) Subscribe(subject string, handler Handler) error {
    tracer := otel.Tracer("cmdb-core/eventbus")
    sub, err := b.nc.Subscribe(subject, func(msg *nats.Msg) {
        ctx := context.Background()
        if msg.Header != nil {
            ctx = otel.GetTextMapPropagator().Extract(ctx, natsHeaderCarrier(msg.Header))
        }
        ctx, span := tracer.Start(ctx, msg.Subject+" receive",
            trace.WithSpanKind(trace.SpanKindConsumer),
            trace.WithAttributes(
                semconv.MessagingSystem("nats"),
                semconv.MessagingDestinationName(msg.Subject),
                attribute.String("messaging.operation", "receive"),
                attribute.Int("messaging.message.payload_size_bytes", len(msg.Data)),
            ),
        )
        defer span.End()

        evt := Event{Subject: msg.Subject, Payload: msg.Data}
        if err := handler(ctx, evt); err != nil {
            span.RecordError(err)
            span.SetStatus(codes.Error, err.Error())
            log.Printf("event handler error [%s]: %v", msg.Subject, err)
        }
    })
    // ... 原 err / mu / subs append 保持不变 ...
}
```

**关键**: handler 收到的 ctx 从 `context.Background()` 变成**提取后的 ctx** —
`handleIncomingEnvelope` 内部的 pgx / 后续 publish 自动挂 consumer span 下,整棵树连起来。
