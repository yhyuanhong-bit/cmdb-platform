# Phase 4.2 — Audit 月度分区与冷归档

> 关联路线图:`/cmdb-platform/docs/reports/audit-2026-04-19/REMEDIATION-ROADMAP.md` § 4.2
> 撰写日期:2026-04-20
> 适用服务:`cmdb-core`(owner of `audit_events`)
> 状态:待实施

---

## 目标

将 `audit_events` 表从当前的单体堆表改造为 Postgres 14+ 声明式(declarative)按月 RANGE 分区表,
并配套每月自动运行的归档任务:把 12 个月(默认)之前的分区从主表 `DETACH`、导出为 Parquet 到 S3
冷存储,最后 `DROP`。目标是把 `audit_events` 的"无上限增长 + 无差别全表 `VACUUM` + 无差别
`DELETE` 抖动"改成"写入走最新月分区、读取绝大多数走最近 2-3 个月分区、历史访问走对象存储"。

```sql
CREATE OR REPLACE FUNCTION prevent_audit_mutation() RETURNS TRIGGER AS $$

    RAISE EXCEPTION 'audit_events table is append-only: % operations are not allowed', TG_OP;
    RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER audit_events_no_delete BEFORE DELETE ON audit_events FOR EACH ROW EXECUTE FUNCTION prevent_audit_mutation();
CREATE TRIGGER audit_events_no_update BEFORE UPDATE ON audit_events FOR EACH ROW EXECUTE FUNCTION prevent_audit_mutation();
```

**注意:append-only trigger 和 `purge_old_audit_events()` 的 `DELETE` 直接冲突** ——
当前函数如果被调用会立刻抛 `audit_events table is append-only: DELETE operations are not allowed`。
这解释了为什么从未见过老数据被清理。该函数在生产环境中实际是死代码。

### 增长估算

可用于估算的 in-repo 信号:

1. `000023` migration 里把 partial index 窗口压到 90 天 —— 作者当时已预判到近 90 天的
   热数据量与 365 天全量差一个数量级,否则不会单列 partial index。
2. 业务分类:登录事件、资产增删改、工单流转、同步事件、告警流转、inventory 任务状态翻转
   共 6 大类,保守估计 10-20 事件/租户/小时 = ~300 事件/天/租户。
3. 10 租户 × 300 事件 × 30 天 ≈ **9 万行/月**。规模扩展到 100 租户 → **90 万行/月**,
   一年 ~10M 行。
4. JSONB `diff` 字段平均 ~2 KB(典型 asset diff ~1-3 KB),单行 ~2.5 KB。10M 行 ≈ **25 GB/年**,
   加索引 ~40 GB/年。

在当前硬件下:
- 单表 40 GB,`VACUUM` 全表扫描 > 30 分钟;`pg_repack` 需要等量磁盘空间。
- `tenant_id, created_at DESC` 复合索引进入 > 2 GB,`Bitmap Heap Scan` 回表抖动明显。
- PITR 恢复点 RTO 被 audit 表放大。
- 没有任何分区/TTL,磁盘压力只增不减。

## 风险 & 触发场景

| 触发场景 | 无 4.2 时的后果 | 严重度 |
|----------|----------------|--------|
| 年增长 10M+ 行 | `VACUUM FULL` 需维护窗口、停写 | HIGH |
| `audit_events` 查询走 seq scan | 审计控制台 > 5s 响应、前端超时 | HIGH |
| 合规审计要求 7 年留存 | 热库无法承载,需冷归档但目前无流水线 | CRITICAL |
| 误用 `purge_old_audit_events()` | 与 append-only trigger 冲突、触发报警 | MEDIUM |
| PITR 恢复 | WAL 中审计写入占比过大,恢复时间倍增 | MEDIUM |
| 对象存储侧缺失 | 一旦热库故障,历史审计永久丢失 | CRITICAL |

## 设计决策

### 选项 1:Postgres 声明式分区(RANGE BY `created_at`,月粒度)— **推荐**

优点:
- 原生、零依赖
- `DETACH PARTITION` 瞬间完成(metadata 操作,不移动数据)
- 查询按 `created_at` 谓词自动做 partition pruning
- 全局索引自动下推到子分区
- `pg_dump --section=data --table=audit_events_2025_04` 按分区备份/导出

缺点:
- 分区创建需脚本化(没有 `pg_cron`/`pg_partman` 的话要自己维护)
- `tenant_id` 没在 partition key,跨分区的单租户查询需要扫多个分区(但每分区有复合索引,可控)

### 选项 2:`pg_partman` 扩展

优点:
- 现成的"每日/每月自动建分区 + 自动 detach + 自动归档"
- retention policy 作为扩展配置项,声明式

缺点:
- 需要 superuser 安装扩展,部分托管 PG 环境不允许
- 增加一个运维目标(版本升级 / 日志 / 报警)
- 本项目并未显式声明依赖 `pg_partman`,新增需过架构评审

### 选项 3:继承式(inherit-based)分区(老派方案)

优点:无

缺点:
- 已被 Postgres 官方文档标记为遗留用法
- CHECK 约束手写、trigger 手写、`INSERT` 需路由
- 与现有 append-only trigger 冲突升级

### 选项 4:TimescaleDB hypertable

优点:
- 项目已有 `timescaledb` 扩展(见 `000009_timescaledb_metrics.up.sql`)
- 自动 chunk + compression

缺点:
- audit 是低写高读/归档型,不是时序指标型,不匹配 hypertable 设计意图
- 引入第二类"特殊表",破坏 ORM / sqlc 假设
- 与 append-only trigger 的兼容性需额外验证

### 决策

**选方案 1(原生声明式 RANGE 分区)**。
`pg_partman` 留作未来优化项,只有当每月建分区 cron 变成维护负担时再引入。
不采用 TimescaleDB,因为 audit 不是时序工作负载。

### 步骤 1 — 创建归档 CLI 骨架(先于 migration,便于回滚时有工具可用)

新增 `/cmdb-platform/cmdb-core/cmd/audit-archive/main.go`:

```go
package main

import (
	"context"
	"flag"
	"fmt"
	"log/slog"
	"os"
	"time"

	"github.com/aws/aws-sdk-go-v2/aws"
	"github.com/aws/aws-sdk-go-v2/config"
	"github.com/aws/aws-sdk-go-v2/service/s3"
	"github.com/jackc/pgx/v5/pgxpool"
)

// audit-archive: detach + export + drop 一批指定月份的 audit 分区
//
//	audit-archive --month 2025-04 --bucket cmdb-audit-archive --retention-months 12
func main() {
	var (
		month     = flag.String("month", "", "YYYY-MM (required)")
		bucket    = flag.String("bucket", "", "S3 bucket for parquet output")
		retention = flag.Int("retention-months", 12, "only archive partitions older than N months")
		dryRun    = flag.Bool("dry-run", false, "plan only, do not mutate DB or S3")
	)
	flag.Parse()

	if *month == "" || *bucket == "" {
		fmt.Fprintln(os.Stderr, "usage: audit-archive --month YYYY-MM --bucket NAME")
		os.Exit(2)
	}

	logger := slog.New(slog.NewJSONHandler(os.Stdout, nil))
	ctx := context.Background()

	pool, err := pgxpool.New(ctx, os.Getenv("DATABASE_URL"))
	if err != nil {
		logger.Error("connect db", "err", err)
		os.Exit(1)
	}
	defer pool.Close()

	awsCfg, err := config.LoadDefaultConfig(ctx)
	if err != nil {
		logger.Error("load aws", "err", err)
		os.Exit(1)
	}
	s3c := s3.NewFromConfig(awsCfg)

	if err := archiveMonth(ctx, logger, pool, s3c, *bucket, *month, *retention, *dryRun); err != nil {
		logger.Error("archive failed", "month", *month, "err", err)
		os.Exit(1)
	}
	logger.Info("archive complete", "month", *month, "dry_run", *dryRun)
	_ = aws.Bool // keep import if unused
	_ = time.Time{}
}
```

核心函数签名稍后在 `archive.go` 里实现(`detach → copy → upload → drop`)。
先有骨架、再补业务逻辑,保证 migration 能回滚。

### 步骤 2 — Migration 000051:建分区父表 + 回填

新文件 `/cmdb-platform/cmdb-core/db/migrations/000051_audit_partitioning.up.sql`:

```sql
-- 000051_audit_partitioning.up.sql
-- 把 audit_events 从单体堆表改造为按 created_at 月分区的声明式 RANGE 分区表。
-- 所有现存行一次性拷贝到"legacy" 分区,然后为当月与未来 3 个月提前创建空分区。

BEGIN;

-- 1. rename 旧表,避免数据丢失
ALTER TABLE audit_events RENAME TO audit_events_legacy;

-- 1a. 必须先删除旧 append-only trigger,否则下一步 DELETE/INSERT 会被拒
DROP TRIGGER IF EXISTS audit_events_no_delete ON audit_events_legacy;
DROP TRIGGER IF EXISTS audit_events_no_update ON audit_events_legacy;

-- 2. 创建分区父表(schema 与旧表完全一致,除了把 PK 从 (id) 改为 (id, created_at)
--    以满足"分区键必须属于 unique 约束"的要求)
CREATE TABLE audit_events (
    id          UUID        NOT NULL DEFAULT gen_random_uuid(),
    tenant_id   UUID        NOT NULL REFERENCES tenants(id),
    action      VARCHAR(50) NOT NULL,
    module      VARCHAR(30),
    target_type VARCHAR(30),
    target_id   UUID,
    operator_id UUID        REFERENCES users(id),
    diff        JSONB,
    source      VARCHAR(20) NOT NULL DEFAULT 'web',
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (created_at);

-- 3. 全局索引(会自动下推到每个子分区)
CREATE INDEX idx_audit_events_tenant_created ON audit_events (tenant_id, created_at DESC);
CREATE INDEX idx_audit_events_target         ON audit_events (target_type, target_id);
CREATE INDEX idx_audit_events_operator       ON audit_events (operator_id);

-- 4. 创建"legacy catch-all"分区:边界覆盖从 epoch 到当前月 1 日 00:00:00 UTC
--    这个分区容纳全部历史数据,只读,后续由归档任务处理
CREATE TABLE audit_events_legacy_partition
    PARTITION OF audit_events
    FOR VALUES FROM (MINVALUE) TO (date_trunc('month', now()));

-- 5. 把旧表数据一次性搬进 legacy 分区
INSERT INTO audit_events_legacy_partition
    (id, tenant_id, action, module, target_type, target_id,
     operator_id, diff, source, created_at)
SELECT id, tenant_id, action, module, target_type, target_id,
       operator_id, diff, source, created_at
FROM audit_events_legacy
WHERE created_at < date_trunc('month', now());

-- 边界行(created_at >= 当月 1 号 00:00)放入当月分区,见第 6 步
-- 先把当月行暂存到 temp 表,避免一会儿建分区时的 range 冲突
CREATE TEMP TABLE audit_events_current_month AS
SELECT * FROM audit_events_legacy WHERE created_at >= date_trunc('month', now());

-- 6. 创建当月 + 未来 3 个月的分区(避免写入时撞到缺失分区)
DO $$
DECLARE
    m date := date_trunc('month', now())::date;
    i int;
    p_name text;
    p_from text;
    p_to   text;
BEGIN
    FOR i IN 0..3 LOOP
        p_name := 'audit_events_' || to_char(m + (i || ' month')::interval, 'YYYY_MM');
        p_from := to_char(m + (i     || ' month')::interval, 'YYYY-MM-DD');
        p_to   := to_char(m + ((i+1) || ' month')::interval, 'YYYY-MM-DD');
        EXECUTE format(
            'CREATE TABLE IF NOT EXISTS %I PARTITION OF audit_events
             FOR VALUES FROM (%L) TO (%L)',
            p_name, p_from, p_to
        );
    END LOOP;
END $$;

-- 7. 把当月数据搬进来
INSERT INTO audit_events SELECT * FROM audit_events_current_month;
DROP TABLE audit_events_current_month;

-- 8. 重新装上 append-only trigger(装在父表上,子分区自动继承)
CREATE TRIGGER audit_events_no_delete
    BEFORE DELETE ON audit_events
    FOR EACH ROW EXECUTE FUNCTION prevent_audit_mutation();
CREATE TRIGGER audit_events_no_update
    BEFORE UPDATE ON audit_events
    FOR EACH ROW EXECUTE FUNCTION prevent_audit_mutation();

-- 9. 删除 000023 引入的 "purge_old_audit_events" 死代码
--    它与 append-only trigger 冲突,分区化后由 audit-archive CLI 替代
DROP FUNCTION IF EXISTS purge_old_audit_events();

-- 10. 删除 000023 引入的 partial index idx_audit_events_recent
--     WHERE 子句用 now() 属于非 IMMUTABLE 表达式,在分区表上无意义
DROP INDEX IF EXISTS idx_audit_events_recent;

-- 11. 旧表保留 7 天以便灾后回滚(下一个 migration 再删)
COMMENT ON TABLE audit_events_legacy IS 'orphan from 000051; safe to drop after 7 days of steady state';

COMMIT;

-- 12. VACUUM 在事务外执行,且走 ANALYZE 来刷新统计
VACUUM ANALYZE audit_events;
```

### 步骤 3 — 完善 `audit-archive` 主逻辑

新增 `/cmdb-platform/cmdb-core/cmd/audit-archive/archive.go`(细节只给骨架,实际实现走
子 PR):

```go
package main

import (
	"context"
	"fmt"
	"log/slog"
	"time"

	"github.com/aws/aws-sdk-go-v2/service/s3"
	"github.com/jackc/pgx/v5/pgxpool"
)

func archiveMonth(ctx context.Context, log *slog.Logger, pool *pgxpool.Pool,
	s3c *s3.Client, bucket, month string, retentionMonths int, dryRun bool) error {

	// 1. parse YYYY-MM,校验 month 至少在 retentionMonths 之前
	t, err := time.Parse("2006-01", month)
	if err != nil {
		return fmt.Errorf("parse month: %w", err)
	}
	cutoff := time.Now().UTC().AddDate(0, -retentionMonths, 0)
	if t.After(cutoff) {
		return fmt.Errorf("month %s newer than retention cutoff %s", month, cutoff.Format("2006-01"))
	}

	partition := "audit_events_" + t.Format("2006_01")

	// 2. 校验分区存在
	var exists bool
	err = pool.QueryRow(ctx,
		`SELECT EXISTS(SELECT 1 FROM pg_class WHERE relname=$1)`,
		partition).Scan(&exists)
	if err != nil {
		return err
	}
	if !exists {
		log.Warn("partition missing, skipping", "partition", partition)
		return nil
	}

	// 3. DETACH CONCURRENTLY(PG14+)— 零锁
	if !dryRun {
		_, err = pool.Exec(ctx,
			fmt.Sprintf("ALTER TABLE audit_events DETACH PARTITION %s CONCURRENTLY", partition))
		if err != nil {
			return fmt.Errorf("detach: %w", err)
		}
	}

	// 4. COPY 出来为 parquet(使用 duckdb 或手写 parquet writer),上传 S3
	//    key = audit/YYYY/MM/tenant=all/audit_events_YYYY_MM.parquet
	//    必须启用 SSE-KMS 和 object lock
	key := fmt.Sprintf("audit/%s/%s/%s.parquet",
		t.Format("2006"), t.Format("01"), partition)
	log.Info("uploading", "bucket", bucket, "key", key, "dry_run", dryRun)

	// 5. 校验 S3 对象存在且 checksum 正确(HEAD + ETag)
	// 6. DROP 分区
	if !dryRun {
		_, err = pool.Exec(ctx, fmt.Sprintf("DROP TABLE %s", partition))
		if err != nil {
			return fmt.Errorf("drop: %w", err)
		}
	}

	return nil
}
```

### 步骤 4 — 每月 cron:建下月分区 + 归档 12 月前分区

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: audit-archive
  namespace: cmdb
spec:
  # 每月 1 号 02:00 UTC 跑一次
  schedule: "0 2 1 * *"
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 5
  concurrencyPolicy: Forbid
  jobTemplate:
    spec:
      backoffLimit: 2
      template:
        spec:
          restartPolicy: OnFailure
          serviceAccountName: audit-archive
          containers:
            - name: archive
              image: ghcr.io/cmdb/audit-archive:latest
              env:
                - name: DATABASE_URL
                  valueFrom: { secretKeyRef: { name: cmdb-db, key: url } }
                - name: AWS_REGION
                  value: us-west-2
              command:
                - /bin/sh
                - -c
                - |
                  set -euo pipefail
                  # 1. 提前建下个月分区(如已存在则 no-op)
                  NEXT_MONTH=$(date -u -d "+1 month" +%Y-%m-01)
                  NEXT_NEXT=$(date -u -d "+2 month" +%Y-%m-01)
                  NEXT_NAME=$(date -u -d "+1 month" +%Y_%m)
                  psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -c "
                    CREATE TABLE IF NOT EXISTS audit_events_${NEXT_NAME}
                    PARTITION OF audit_events
                    FOR VALUES FROM ('${NEXT_MONTH}') TO ('${NEXT_NEXT}');"

                  # 2. 归档 13 个月前分区(保留窗口 12 月 + 本月)
                  OLD_MONTH=$(date -u -d "-13 month" +%Y-%m)
                  /usr/local/bin/audit-archive \
                      --month "$OLD_MONTH" \
                      --bucket cmdb-audit-archive \
                      --retention-months 12
```

```sql
CREATE EXTENSION IF NOT EXISTS pg_cron;

CREATE OR REPLACE FUNCTION ensure_next_audit_partition() RETURNS void
LANGUAGE plpgsql AS $$
DECLARE
    next_start date := date_trunc('month', now() + interval '1 month')::date;
    next_end   date := date_trunc('month', now() + interval '2 month')::date;
    p_name     text := 'audit_events_' || to_char(next_start, 'YYYY_MM');
BEGIN
    EXECUTE format(
        'CREATE TABLE IF NOT EXISTS %I PARTITION OF audit_events
         FOR VALUES FROM (%L) TO (%L)',
        p_name, next_start, next_end);
END $$;

SELECT cron.schedule(
    'audit-next-partition', '0 1 25 * *',
```

## 回滚方案

**情况 1:migration 失败(在本机 / staging)**

直接运行 `golang-migrate down 000051`。down migration 把数据从分区表 `INSERT ... ON
CONFLICT DO NOTHING` 回 `audit_events_legacy`,然后 rename 回原名。

**情况 2:上线后发现查询抖动,需要临时回滚**

1. 先停 `audit-archive` CronJob:`kubectl -n cmdb suspend cronjob/audit-archive`
2. 对已被归档到 S3 的月份,**不要**尝试重挂回主表,否则破坏 append-only 属性
3. 对仍 attached 的分区,如果需要合并回单体,执行:
   ```sql
   BEGIN;
   -- 建新单体表
   CREATE TABLE audit_events_merged (LIKE audit_events INCLUDING DEFAULTS INCLUDING CONSTRAINTS);
   INSERT INTO audit_events_merged SELECT * FROM audit_events;
   DROP TABLE audit_events CASCADE;
   ALTER TABLE audit_events_merged RENAME TO audit_events;
   -- 重建 trigger/index
   COMMIT;
   ```
4. 从 `audit_events_legacy`(若仍保留)拷回缺失行

**情况 3:S3 归档失败,分区已 DETACH 但未 DROP**

`audit-archive` CLI 的设计是"DETACH → upload → verify → DROP",任意一步失败都不会进到下一步。
DETACH 后的孤儿分区可以:

```sql
-- 把孤儿分区重新挂回父表
ALTER TABLE audit_events ATTACH PARTITION audit_events_2025_03
    FOR VALUES FROM ('2025-03-01') TO ('2025-04-01');
```

CLI 需要在错误退出前把状态打印到 stderr + Prometheus counter,便于人工处置。
