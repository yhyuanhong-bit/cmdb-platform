## 一、設計背景與核心問題

### 1.1 企業面臨的痛點

```
              傳統 IT 基礎設施管理
                     │
    ┌────────────────┼────────────────┐
    ▼                ▼                ▼
  資產資料散落      維運流程斷裂      決策缺乏數據
  ├─ Excel 表格    ├─ 口頭交接      ├─ 不知道有多少資產
  ├─ 多套系統不同步  ├─ 工單紙本      ├─ 不知道哪台快壞
  ├─ 誰改了什麼不知道 ├─ 審批靠 email  ├─ 盤點靠人工數
  └─ 新設備沒人登記   └─ SLA 沒人追   └─ 合規報告湊不齊
```

### 1.2 IronGrid 要解決什麼

**一個平台統一管理：資產在哪、狀態如何、誰在維護、什麼時候會壞。**

核心設計目標：
1. **唯一資料來源 (SSOT)**：所有資產資訊只有一份，自動同步
2. **流程閉環**：從發現設備 → 入庫 → 維護 → 報廢，全程追蹤
3. **智能預警**：AI 預測故障，而不是壞了才知道
4. **分散式部署**：總部 + 遠端機房都能用，斷網也能工作

## 二、架構設計思路

### 2.1 為什麼選擇這個架構

#### 問題 → 方案映射

| 問題 | 架構決策 | 理由 |
|------|---------|------|
| 資料散落多處 | **集中式 CMDB + 欄位權威矩陣** | 一個 DB 當權威，誰能改哪個欄位有規則 |
| 設備型號多、欄位差異大 | **JSONB attributes 彈性欄位** | 伺服器和交換機欄位不同，用 JSONB 不用改 schema |
| 位置層級複雜 | **PostgreSQL ltree** | 7 層階層（國家→區域→城市→園區→IDC→模組→機房），原生遞迴查詢 |
| 需要即時反應 | **NATS JetStream 事件驅動** | 工單完成 → 自動解決告警，不用輪詢 |
| 遠端機房網路不穩 | **Edge 離線同步 + NATS leafnode** | 斷網本地存，恢復自動同步 |
| 時序指標量大 | **TimescaleDB hypertable** | 同一個 PostgreSQL，不需要額外部署 InfluxDB |
| 需要 AI 預測 | **MCP Server + 多 Provider** | 接 Dify/Claude/OpenAI，可插拔切換 |
| 多語言用戶 | **i18n + 3 語言** | 台灣（繁中）、大陸分公司（簡中）、外籍人員（英文） |

### 2.2 核心設計模式

#### 模式 1：事件驅動的跨模組協作

```
用戶建立工單 → API 寫入 DB → 發布 NATS 事件
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
              WebSocket         Workflow          Webhook
              推送前端          跨模組反應         推送外部
              即時更新          (工單完成→         (Slack
                               解決告警→          通知)
                               通知責任人)
```

**為什麼不用直接函數調用？** 因為模組之間不應該互相依賴。工單模組不需要知道告警模組的存在，它只需要發布「工單完成了」這個事件。

#### 模式 2：唯一真相來源 (SSOT) 防護

```
                ┌─ 品質閘門（分數 < 40 拒絕）
                │
外部資料 ──→ 管線 ──→ 權威檢查 ──→ CMDB
                │        │
                │   欄位優先級矩陣
                │   IPMI:100 > SNMP:80 > 手動:50
                │
                └─ 衝突佇列（7 天 SLA）
```

#### 模式 3：分層有序同步

```
Edge 恢復連線，需要同步 500 筆變更
    │
    ▼ 按依賴層級處理
Layer 0: 位置、使用者、角色         ← 先同步（無依賴）
Layer 1: 機櫃、資產                 ← 依賴 L0
Layer 2: 機櫃插槽、告警             ← 依賴 L1
Layer 3: 工單、盤點任務             ← 依賴 L2
Layer 4: 操作日誌、稽核             ← 依賴 L3

每層完成才進下一層，同層內並行處理
```

### 2.3 安全設計

```
請求進入
  │
  ▼
JWT 認證（15 分鐘 Token + 7 天 Refresh）
  │
  ▼
RBAC 授權（角色 → 資源 + 動作，Redis 快取 5 分鐘）
  │
  ▼
業務邏輯執行
  │
  ▼
稽核記錄（98% 覆蓋，append-only，DB trigger 防篡改）
  │
  ▼
事件發布（NATS → WebSocket / Webhook / Workflow）
```

## 五、關鍵設計決策清單

| # | 決策 | 選項 | 選擇 | 理由 |
|---|------|------|------|------|
| 1 | 後端語言 | Go / Java / Node.js | **Go** | 效能高、部署簡單（單一 binary）、適合基礎設施工具 |
| 2 | 資料庫 | PostgreSQL / MySQL / MongoDB | **PostgreSQL + TimescaleDB** | 關係型 + 時序一體，ltree 支援階層 |
| 3 | 訊息佇列 | NATS / RabbitMQ / Kafka | **NATS JetStream** | 輕量、leafnode 聯邦（Edge 同步）、JetStream 持久化 |
| 4 | ORM vs SQL | GORM / sqlc / raw SQL | **sqlc** | 類型安全、效能好、SQL 可控可審計 |
| 5 | 前端框架 | React / Vue / Angular | **React 19** | 生態最大、TypeScript 支援好 |
| 6 | 狀態管理 | Redux / Zustand / Context | **React Query + Zustand** | Query 管 API 快取、Zustand 管 Auth |
| 7 | CSS | Tailwind / CSS Modules / styled | **Tailwind CSS** | 快速開發、一致性好 |
| 8 | 位置階層 | 遞迴 CTE / ltree / Nested Set | **ltree** | PostgreSQL 原生、索引高效 |
| 9 | 認證 | JWT / Session / OAuth | **JWT + Refresh Token** | 無狀態、適合分散式 |
| 10 | Edge 同步衝突 | LWW / CRDT / Manual | **分層策略** | 不同欄位不同策略（見 RFC） |
| 11 | 工單衝突 | LWW / Manual / 雙維度 | **雙維度狀態模型** | execution + governance 分離，零衝突 |
| 12 | 初始同步 | pg_dump / API 全量 / 自動快照 | **自動快照** | Edge 部署零人工操作 |
| 13 | 離線窗口 | 7 天 / 14 天 | **14 天** | 2.6MB 佇列量，成本極低 |
| 14 | 資料品質 | 僅評分 / 閘門阻擋 | **閘門 + 評分** | < 40 分拒絕入庫 |
| 15 | 同步重試 | 單一佇列 / 實體類型佇列 / 分層 | **5 層有序佇列** | 依賴安全、無死鎖 |

## 七、AI 如何加速了這個過程

### 7.1 設計階段

| 傳統方式 | AI 輔助方式 | 提速 |
|---------|-----------|------|
| 架構師畫圖 + 寫文件（1-2 週） | 對話式設計，即時產出 RFC + 架構圖（2 小時） | **40x** |
| 技術選型調研（3-5 天） | AI 列出選項 + 優缺點 + 推薦（10 分鐘） | **30x** |

## 八、結論

IronGrid CMDB 平台在 12 天內從零建立了一套包含 **90+ API、52 頁面、32 表、3 語言、Edge 離線同步**的企業級平台。這不是一個 Demo 或 POC——它有完整的認證授權、稽核軌跡、資料品質閘門、分散式同步、AI 預測整合和 E2E 測試。

**核心啟示**：AI 沒有降低對工程能力的要求，反而提高了——因為你需要更快地做出更多正確的決策。AI 把「寫程式碼」的時間壓縮到了接近零，但「想清楚要寫什麼」的時間一分鐘都不能少。
