> 日期：2026-04-14
> 適用：IronGrid CMDB Platform
> 目標讀者：管理層 + 維運團隊 + 開發團隊

---

## 一、問題現狀

### 1.1 核心痛點

資產位置資訊與實際不符，是 CMDB 最常見的**資料腐化問題**。

```
CMDB 記錄：伺服器 A 在 3 樓 RACK-A01 第 12U
實際情況：伺服器 A 上週被搬到 5 樓 RACK-C03 第 8U

結果：
  ├─ 維運人員到 3 樓找不到機器 → 浪費 30 分鐘
  ├─ 告警發生時不知道機器在哪 → 延長故障恢復時間
  ├─ 容量規劃失準 → 3 樓以為滿了其實不滿，5 樓以為空的其實滿了
  └─ 合規稽核不過 → 資產清冊與現場不符
```

### 1.2 根因分析

| 根因 | 占比（估） | 說明 |
|------|-----------|------|
| **搬遷後不更新 CMDB** | 60% | 工程師搬完設備懶得改系統，覺得是額外工作 |
| **臨時借調不記錄** | 20% | 「借兩天就還」結果一借三個月 |
| **初始資料就不對** | 10% | 錄入時位置就填錯了 |
| **批量遷移遺漏** | 10% | 搬了 50 台只更新了 45 台 |

**本質問題**：更新 CMDB 對搬設備的人沒有好處，只是額外負擔。

## 二、現有平台能力評估

### 2.1 已具備的能力

| 能力 | 說明 | 有效性 |
|------|------|--------|
| 資產位置欄位 | `assets.rack_id` + `assets.location_id` | 有，但依賴人工更新 |
| U 位管理 | `rack_slots` 表（start_u, end_u, side） | 有，但搬遷後需手動改 |
| 網路連線記錄 | `rack_network_connections` 表 | 有，但搬遷後不自動更新 |
| 稽核軌跡 | 位置變更記錄到 `audit_events` | 有，事後可追溯 |
| API 更新 | `PUT /assets/{id}` 可改位置 | 有，但沒人用 |
| 盤點功能 | Excel 匯入 + 現場掃描 + 差異處理 | 有，但是事後發現 |
| 工單系統 | 維護工單 + 審批 + SLA | 有，但沒有搬遷專用類型 |
| BIA 影響分析 | 級聯衝擊分析 | 有，但搬遷前不會自動觸發 |

### 2.2 缺失的能力

| 缺失 | 影響 |
|------|------|
| 搬遷工單流程 | 搬遷無追蹤，不知道誰批的、誰搬的、何時完成 |
| 設備搬遷中狀態 | 拔掉到插好之間，CMDB 不知道設備在路上 |
| QR 掃碼快速更新 | 更新位置需要登入系統手動改，太麻煩 |
| 網路拓撲自動偵測 | 無法靠交換機 MAC 表自動發現位置變化 |
| 搬遷前影響評估 | 搬關鍵設備沒有自動觸發 BIA 影響分析 |
| 位置異常告警 | CMDB 記錄與實際不符時沒有主動告警 |

## 三、高速盤點模組現狀

### 3.1 流程概覽

```
┌─────────────────┬─────────────────┬─────────────────┐
│  ① Excel 匯入    │  ② 現場掃描      │  ③ 差異處理      │
│                  │                  │                  │
│ 上傳預期清單      │ 逐台掃描確認      │ 逐筆解決差異      │
│ 系統比對資產庫    │ 機櫃方格圖即時    │ 驗證/清除/登記    │
│ 三級 fallback    │ 更新掃描進度      │ >5筆自動建工單    │
│                  │                  │                  │
│ serial_number    │ ■ scanned       │ ├─ verify ✓     │
│ → asset_tag      │ ■ pending       │ ├─ clear  ✓     │
│ → property_no    │ ■ not_started   │ ├─ register ✓   │
│ → control_no     │                  │ └─ findings ⚠   │
│ → missing        │                  │                  │
└─────────────────┴─────────────────┴─────────────────┘
```

### 3.2 資產比對邏輯

```
Excel 一筆資料
  │
  ├─ 1. serial_number + asset_tag 聯合查詢 → 找到 ✓
  │
  ├─ 2. property_number 查詢 → 找到 ✓
  │
  ├─ 3. control_number 查詢 → 找到 ✓
  │
  └─ 全都沒找到 → 標記 missing
```

支援中英文表頭：`Asset Tag` / `资产编号` / `資產編號`

### 3.3 差異處理

| 動作 | 新狀態 | 場景 |
|------|--------|------|
| verify | scanned | 現場確認無誤，帳實相符 |
| clear | scanned | 設備已報廢/移除，清除記錄 |
| register | scanned | 發現未登記設備，新增到 CMDB |
| add_findings | discrepancy | 需要調查，保持差異狀態 |

差異超過 5 筆 → 自動建立 high 優先級維護工單。

### 3.4 盤點模組已知問題

| # | 問題 | 影響 |
|---|------|------|
| 1 | 匯入進度條硬編碼 `142/150` | 使用者看到假進度 |
| 2 | 匯入錯誤列表硬編碼 3 筆 | 不反映真實匯入問題 |
| 3 | QR 掃描未實作 | 只能手動輸入，效率低 |
| 4 | 匯出報告繞過認證 | 直接 `fetch()` 不帶 token |
| 5 | 機櫃方格圖無法點擊到特定機櫃 | 所有 tile 都導向同一頁面 |
| 6 | 盤點只報告差異不更新 CMDB | 發現位置不對但不自動修正 |

### 4.1 架構總覽

```
      │ 有工單   │     │ 自動更新  │     │ 位置變化  │     │ 主動告警  │
      │         │     │          │     │          │     │          │
      │ 管理制度 │     │ QR Code  │     │ MAC 表    │     │ 持續驗證  │
      │ + 審批   │     │ + 自動    │     │ + LLDP   │     │ + 升級    │
      └────┬────┘     └────┬─────┘     └────┬─────┘     └────┬─────┘
           │               │               │               │
           ▼               ▼               ▼               ▼
        減少 50%         減少 80%         減少 95%          趨近 0%
        位置錯誤          位置錯誤          位置錯誤          位置錯誤
```

### 4.2 第一道防線：制度層 — 搬遷必須有工單

**目標**：從源頭堵住「搬了不說」的問題。

#### 搬遷工單流程

```
① 發起搬遷申請
   ├─ 選擇設備（支援多台批量）
   ├─ 指定：從哪個機櫃/U位 → 到哪個機櫃/U位
   ├─ 排定搬遷時間窗口
   └─ 系統自動：
       ├─ 檢查目標機櫃是否有空間（U位衝突檢測）
       ├─ 檢查目標機櫃電力是否足夠
       └─ BIA 級聯分析 → 列出受影響的下游系統

② 審批
   ├─ 普通設備（BIA Normal/Minor）→ ops-admin 審批
   ├─ 重要設備（BIA Important）→ ops-admin + 部門主管
   └─ 關鍵設備（BIA Critical）→ 需要 CTO/CIO 審批
       └─ 自動通知所有受影響系統的負責人

③ 執行
   ├─ 設備狀態：operational → relocating
   ├─ 舊位置：斷開網路、釋放 U 位
   ├─ 運輸中：位置 = in_transit
   └─ 新位置：插線、分配 U 位

④ 確認
   ├─ 掃描新機櫃 QR → 掃描設備 QR → CMDB 自動更新
   ├─ 設備狀態：relocating → operational
   └─ 稽核記錄完整搬遷軌跡

⑤ 驗證
   └─ 下次盤點自動驗證新位置正確
```

#### 搬遷期間的設備狀態

```
operational ──→ relocating ──→ in_transit ──→ relocating_in ──→ operational
                    │                                              │
                    └── rejected（審批拒絕）                         │
                                                                   │
                                              CMDB 自動更新位置 ───┘
```

**實施要求**：
- 管理制度：機房出入需出示搬遷工單編號
- 門禁系統聯動（如有）：無工單不開門
- KPI 追蹤：搬遷工單閉環率

### 4.3 第二道防線：操作層 — QR 掃碼即更新

**目標**：把更新 CMDB 的成本降到 6 秒。

#### QR Code 體系

```
機櫃門上                        設備上
┌──────────────┐              ┌──────────────┐
│ ▄▄▄▄▄ ▄▄▄▄▄ │              │ ▄▄▄▄▄ ▄▄▄▄▄ │
│ █ ▄▄▄ █ █▄█ │              │ █ ▄▄▄ █ █▄█ │
│ █ ███ █ ▄▄▄ │              │ █ ███ █ ▄▄▄ │
│ ▀▀▀▀▀ ▀▀▀▀▀ │              │ ▀▀▀▀▀ ▀▀▀▀▀ │
│              │              │              │
│ RACK-A01     │              │ SRV-PROD-001 │
│ Room 4A      │              │ SN-DELL-001  │
└──────────────┘              └──────────────┘

內容：                         內容：
{                              {
  "t": "rack",                   "t": "asset",
  "id": "RACK-A01",              "tag": "SRV-PROD-001",
  "loc": "Room-4A"               "sn": "SN-DELL-001"
}                              }
```

#### 掃碼操作流程

```
工程師搬完設備，掏出手機
  │
  ▼
掃一下新機櫃門上的 QR（3 秒）
  → 系統知道：目標位置 = RACK-B03, Room-5A
  │
  ▼
掃一下設備上的 QR（3 秒）
  → 系統知道：設備 = SRV-PROD-001
  │
  ▼
系統自動完成（0 秒）：
  ├─ 更新 assets.rack_id → RACK-B03
  ├─ 更新 assets.location_id → Room-5A
  ├─ 釋放舊機櫃 RACK-A01 的 U 位
  ├─ 分配新機櫃 RACK-B03 的 U 位
  ├─ 更新網路連線記錄
  ├─ 稽核日誌：搬遷，操作人=XXX，時間=YYY
  ├─ 通知相關人員
  └─ 如果有對應搬遷工單 → 自動標記完成
```

**比打開 Excel 改一行還快。**

#### 技術實作

- 不需要原生 App，瀏覽器內建相機即可（`html5-qrcode` 函式庫）
- QR Code 由系統在資產/機櫃入庫時自動生成
- 支援批量列印 QR 標籤（匯出 PDF）

### 4.4 第三道防線：技術層 — 網路拓撲自動偵測

**目標**：即使人不掃碼，系統也能自動發現位置變化。

#### 偵測原理

```
每台設備有唯一的 MAC 位址
  │
交換機記錄「哪個 MAC 在哪個端口」（MAC Address Table）
  │
每個交換機端口對應一個機櫃位置
  │
MAC 出現在不同交換機端口 = 設備換了位置
```

#### 偵測流程

```
SNMP 每 5 分鐘掃描所有交換機 MAC 表
  │
  ├─ 上次：SRV-001 的 MAC 在 Switch-A01 Port 12（RACK-A01）
  ├─ 這次：SRV-001 的 MAC 在 Switch-B03 Port 8（RACK-B03）
  │
  ▼
位置變化偵測觸發
  │
  ├─ 查詢：有對應的搬遷工單嗎？
  │   │
  │   ├─ 有 → 自動確認搬遷完成 ✓
  │   │       更新 CMDB 位置
  │   │       關閉搬遷工單
  │   │
  │   └─ 沒有 → 生成告警：未授權位置變更 ⚠️
  │              通知機房管理員
  │              通知資產負責人
  │              72 小時未處理 → 自動升級
  │
  └─ 記錄到 audit_events：
      action=location_change_detected
      source=snmp_mac_table
```

#### 偵測能力矩陣

| 偵測方式 | 能偵測什麼 | 準確度 | 已有基礎 |
|---------|-----------|--------|---------|
| **交換機 MAC 表** | 設備插到不同交換機端口 | 高 | SNMP collector 已有 |
| **LLDP/CDP** | 設備的鄰居關係變化 | 高 | 需擴展 SNMP OID |
| **IPMI BMC** | BMC IP/子網變化 | 中 | IPMI collector 已有 |
| **SSH 偵測** | 主機名/IP 變化 | 中 | SSH collector 已有 |
| **盤點核對** | 預期 vs 實際位置 | 100% | 盤點模組已有 |

### 4.5 第四道防線：稽核層 — 持續驗證與告警

**目標**：讓位置錯誤無法被忽視。

#### 每日自動驗證

```
每日凌晨 2:00 自動執行
  │
  ▼
比對三個資料來源：
  ├─ CMDB 記錄：SRV-001 在 RACK-A01
  ├─ 交換機 MAC 表：SRV-001 的 MAC 在 RACK-B03
  └─ 最近盤點結果：SRV-001 上次掃描在 RACK-B03
  │
  ▼
不一致 → 位置異常告警
  │
  ├─ severity = warning（一般設備）
  ├─ severity = critical（BIA Critical 設備）
  │
  ├─ 通知：
  │   ├─ 資產負責人
  │   ├─ 機房管理員
  │   └─ 如果是 Critical → 通知 ops-admin
  │
  └─ 升級機制：
      ├─ 24h 未處理 → 重發通知
      ├─ 72h 未處理 → 升級到部門主管
      └─ 7 天未處理 → 進入品質分數懲罰（一致性扣分）
```

#### 品質分數聯動

```
資產位置與實際不符
  │
  ▼
品質掃描 → 一致性維度扣 50 分
  │
  ▼
資產品質分數下降
  │
  ▼
進入「最差資產 Top 10」名單
  │
  ▼
部門 KPI 受影響
  │
  ▼
倒逼相關人員處理
```

| 合規稽核通過率 | 風險 | 有信心 |

## 七、與平台其他模組的整合

### 7.1 搬遷流程整合圖

```
搬遷申請
  │
  ├─→ 維運模組：自動建立 type=relocation 工單
  │   └─ SLA：Critical 4h / High 8h / Medium 24h
  │
  ├─→ BIA 模組：GetImpactedAssessments(asset_id)
  │   ├─ Critical → 強制審批 + 全員通知
  │   ├─ Important → ops-admin 審批
  │   └─ Normal → 自動批准
  │
  ├─→ 拓撲模組：
  │   ├─ CheckSlotConflict(目標機櫃, 目標U位)
  │   ├─ 計算目標機櫃剩餘容量
  │   └─ 更新 rack_slots + network_connections
  │
  ├─→ 監控模組：
  │   ├─ 搬遷期間：靜音該設備的告警（避免誤報）
  │   └─ 搬遷完成：恢復監控
  │
  ├─→ 盤點模組：
  │   ├─ 搬遷完成 → 掃碼確認 → 自動盤點
  │   └─ 舊位置下次盤點標記「已搬走」
  │
  ├─→ 品質模組：
  │   └─ 位置變更後重新計算一致性分數
  │
  ├─→ 通知模組：
  │   ├─ 通知搬遷申請人
  │   ├─ 通知目標機房管理員
  │   └─ 通知受影響系統負責人
  │
  └─→ 稽核模組：
      └─ 記錄完整搬遷軌跡
          from_rack / to_rack / operator / approver / timestamp
```

### 7.2 Edge 同步場景

```
Edge 節點的機房人員搬遷設備
  │
  ├─ 在線模式：即時同步到 Central
  │
  └─ 離線模式：
      ├─ 本地 CMDB 即時更新
      ├─ 搬遷記錄存入 NATS 本地佇列
      └─ 恢復連線後自動同步到 Central
          └─ execution_status 由 Edge 控制（零衝突）
```

## 八、結論

### 核心觀點

1. **位置錯誤是管理問題，不只是技術問題**。最有效的第一步是制定搬遷 SOP 並嚴格執行，成本為零。

2. **降低更新成本是關鍵**。從「登入系統手動改」降低到「掃碼 6 秒」，工程師自然會配合。

3. **自動偵測是最終防線**。即使人為疏忽，SNMP MAC 表偵測 + 持續驗證也能在 5 分鐘內發現。

4. **四道防線逐步推進**，不需要一步到位。短期靠制度（0 成本），中期靠工具（3,250 元），長期靠自動化。
