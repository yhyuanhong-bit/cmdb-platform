## 决策

### D1: Service 实体边界

**决策**：选 **A — Business Service**（业务功能）

**含义**：
- `services` 表代表用户可见的业务功能（"在线下单"、"支付"、"客户登入"），不是技术组件
- 与 `bia_assessments` 用 FK 关联（`service_id`），保留现有 BIA 数据
- 一个 asset 能属于多个 service（N:M `service_assets`）
- service 之间能互相依赖（推迟到 Wave 6）

**反对意见**：无

**后续可推翻条件**：第一个客户面谈反馈说错位

**理由**：单人团队 Warn 等于没做；漂移持续累积才是最大成本

### D6: Predictive AI 模块

**决策**：**Phase 1（架构修 + 数据 pipeline + Health Score 重命名 + LLM-RCA RAG）**

**含义**：
- `prediction_models` 表加 tenant_id（修 datamodel agent flagged 的安全问题）
- 把现有「RUL = min(warranty, lifespan)」公式重命名为 "Asset Health Score"，不再叫 prediction
- 现有 LLM-RCA 增强：加 vector DB 检索增强（RAG）历史 incident
- 同时建立 metric collection pipeline 让数据自然累积
- 不做真 ML 训练（数据不够 — 当前只有 20 台 asset 有 metrics + 3 天历史）
- 数据成熟（3-6 个月后）再启动 Phase 2 训练 XGBoost 失败预测模型

**理由**：实测 DB 显示
- 795 active assets，仅 20 (2.5%) 有 metrics
- 3 天 metric 历史 (4/3-4/6)
- 只有 2 已解决告警 + 8 incident 工单 = 无法做 supervised learning

### D7: Energy 模块

**决策**：**Phase 1 + Phase 2**（6 周）

**Phase 1 (Week 1-4)**：
- 3 大 PDU 厂商 SNMP collector：APC、Schneider Electric、Eaton
- UPS collector：APC SmartUPS、Eaton 9PX/9SX
- power_events 表 + 阈值告警
- Dashboard / EnergyMonitor 4 个 phase-3.10 占位接真数据

**Phase 2 (Week 5-6)**：
- PUE (Power Usage Effectiveness) 计算 + 趋势
- 容量预测（Prophet 统计预测，不是 ML）
- 多厂商扩展：Vertiv、CyberPower、ABB
- 碳足迹月报
- TimescaleDB continuous aggregates 支持长周期 trending

**前提**：项目负责人需要在 Week 4 末提供至少 1 台真实 PDU/UPS 设备做集成测试
